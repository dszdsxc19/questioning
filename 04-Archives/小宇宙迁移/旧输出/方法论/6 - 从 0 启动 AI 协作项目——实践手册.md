# 从 0 启动 AI 协作项目——实践手册

> 前五篇文章解决的是"为什么这样做"。这篇解决的是"具体怎么做"。
> 它是 [[5 - 从需求到上线——完整的系统设计方法论]] 和 [[4 - 为 AI Agent 构建开发环境、约束和反馈系统]] 的实践落地版本。

---

## 三个反直觉的事实

在开始之前，先接受三个和直觉相反的事实：

**① 文档在代码之前**
不是先写代码再补文档，是文档写完之后 Agent 才开始写代码。文档是 Agent 的执行依据，不是项目的"附属品"。

**② Harness 在设计之后**
很多人以为"环境搭建"是第一步——建好 monorepo、配好 lint、写好 `ARCHITECTURE.md`，然后开始需求分析。这是错的。`ARCHITECTURE.md` 里写的是架构决策，而架构在系统设计阶段才产出。**Harness 是设计的"编译结果"，不是项目的"起点"。**

**③ 开始前 ≠ 第一阶段**
`git init` + 安装工具，15 分钟搞定，不是一个正式阶段，不产出任何设计决策。

## 全流程总览

```
[开始前 · 15分钟]
git init + 安装工具 + 最简 CLAUDE.md
         ↓
[第 1 阶段] 产品阶段           → docs/PRD.md
         ↓
[第 2 阶段] 系统设计           → docs/SYSTEM_DESIGN.md
                               → docs/TECH_SELECTION.md  ← 决定后续 monorepo 结构
                               → docs/DATA_MODEL.md
         ↓
[第 3 阶段] 详细设计           → docs/API_SPEC.md
                               → docs/TEST_STRATEGY.md
         ↓
[第 4 阶段] Agent Harness 构建 ← 真正的"环境搭建"在这里
         │  按 TECH_SELECTION.md 建 monorepo 包结构
         │  ARCHITECTURE.md（设计文档的 Agent 可读版本）
         │  AGENTS.md + RULES.md
         └  CI/CD 自动验证管线
         ↓
[第 5 阶段] 实现（Agent 主导）
         你 = 审核者，不是执行者
```

## Harness Skills 套件——流程的导航与守护

> 光有流程图不够。你需要一套能随时回答"我在哪、该做什么、有没有走偏"的工具。
> 这就是 harness skills 三件套的作用——它们不是替你做决策，而是让你随时能看清自己在流程中的位置。

### 三个 Skill 的分工

```
/harness          ← 导航仪。随时可查：我在哪？下一步做什么？
/harness-next     ← 推进器。验证当前阶段产物 → 更新进度 → 创建下阶段模板
/harness-check    ← 守护者。脚本检测文件系统，客观报告是否跳步或过早实现
```

**关键设计原则：`/harness-check` 不靠 AI 猜测，靠 bash 脚本检测文件是否真实存在。** 这样偏离检测的结论是客观的，不受上下文干扰。

### 安装方式

把以下命令在项目根目录运行，安装到项目的 `.claude/skills/` 下：

```bash
# 三个 skill 一起安装（来自 Claudian 内置 harness 套件）
npx skills add .claude/skills/harness       --skill harness
npx skills add .claude/skills/harness-next  --skill harness-next
npx skills add .claude/skills/harness-check --skill harness-check
```

安装后会自动在项目根目录创建 `PROGRESS.md`（如果不存在）。

### 典型使用场景
	
**项目刚启动，不知道从哪开始：**
```
你：/harness
→  检测到没有 PROGRESS.md，询问项目名称，初始化进度文件
→  提示：从第1阶段开始，运行 /harness-next 进入产品阶段
```

**随时查看进度：**
```
你：我现在到哪了？
→  /harness 自动触发
→  显示：📍 当前阶段：系统设计阶段（第 2 / 6 阶段）
          已完成 ✅ 产品阶段 → docs/PRD.md
          进行中 🟡 需要产出：SYSTEM_DESIGN.md / TECH_SELECTION.md / DATA_MODEL.md
          💡 完成后运行 /harness-next 推进
```

**当前阶段写完，推进到下一阶段：**
```
你：系统设计完成了，进入详细设计
→  /harness-next 触发
→  脚本验证：SYSTEM_DESIGN.md ✅  TECH_SELECTION.md ✅  DATA_MODEL.md ❌ 不存在
→  停止：DATA_MODEL.md 缺失，无法推进
→  你补写 DATA_MODEL.md 后再次运行 → 验证通过 → 更新 PROGRESS.md → 创建 API_SPEC.md 等模板
```

**担心走偏了：**
```
你：帮我检查一下有没有偏离路线
→  /harness-check 触发
→  脚本扫描文件系统，输出：
   🚨 规则3 违反：ARCHITECTURE.md 存在，但 API_SPEC.md 缺失
      原因：Harness 是设计的「编译结果」，API_SPEC 完成前 ARCHITECTURE.md 无法写准确
      建议：先完成 docs/API_SPEC.md，再重新审视 ARCHITECTURE.md 内容
```

### 偏离检测的五条规则

`/harness-check` 会自动检测以下前置条件违规：

| 规则 | 检测逻辑 | 严重程度 |
|------|---------|---------|
| 规则1 | `SYSTEM_DESIGN.md` 存在 但 `PRD.md` 不存在 | 🚨 严重 |
| 规则2 | `API_SPEC.md` 存在 但 `SYSTEM_DESIGN.md` 不存在 | 🚨 严重 |
| 规则3 | `ARCHITECTURE.md` 存在 但 `API_SPEC.md` 不存在 | 🚨 严重 |
| 规则4 | `AGENTS.md` 或 `RULES.md` 存在 但 `SYSTEM_DESIGN.md` 不存在 | 🚨 严重 |
| 规则5 | 发现实现代码（`.ts/.py/.go`）但 `API_SPEC.md` 不存在 | 🚨 严重 |

> **这五条规则对应的是流程中最常见的跳步模式**——有了它们，"感觉哪里不对"就不再是感觉，而是可检测的客观事实。

### PROGRESS.md 格式

每个项目根目录维护一个 `PROGRESS.md`，是三个 skill 共同读写的状态文件：

```markdown
---
project: "我的项目"
current_stage: 2
total_stages: 6
---

| # | 阶段名称 | 状态 | 已完成产物 |
|---|---------|------|-----------|
| 1 | 产品阶段 | ✅ 完成 | `docs/PRD.md` |
| 2 | 系统设计阶段 | 🟡 进行中 | |
| 3 | 详细设计阶段 | ⏳ 未开始 | |
| 4 | Harness 构建 | ⏳ 未开始 | |
| 5 | 实现阶段 | ⏳ 未开始 | |
| 6 | 部署运维 | ⏳ 未开始 | |
```

`/harness-next` 推进时自动更新这张表，你也可以手动修改。

## 开始前（15 分钟，不是一个阶段）

只做三件事，不做决策：

**① 初始化仓库**

```bash
mkdir my-project && cd my-project
git init
mkdir docs
```

**② 安装 Claude Code + MCP 工具**

这是工具，不是 Harness。随时需要，第一天装好：
- `filesystem` MCP — 文件读写
- `git` MCP — 版本管理
- `browser` MCP — 后期 UI 验证用

**③ 写最简 CLAUDE.md**

```markdown
# 项目名称

一句话描述这是什么项目。

## 当前状态
正在产品设计阶段，尚未有代码。
```

就这些。**不要**在这一步写架构约束、lint 规则、模块说明——你还不知道这些。

## 第 1 阶段：产品阶段

> **这一阶段完全不谈技术。**

只关心一件事：用户的问题是什么，怎么解决？

**产出：`docs/PRD.md`**

```markdown
# PRD: [产品名称]

## 产品目标（一句话）
帮 [谁] 解决 [什么问题]。

## 目标用户
- 用户画像
- 核心使用场景（3-5 个）

## 功能优先级

| 优先级 | 功能 | 说明 |
|--------|------|------|
| P0     | ...  | 核心流程，MVP 必须有 |
| P1     | ...  | 重要，但不是 MVP |
| P2     | ...  | 未来迭代 |

## 用户流程（产品视角，不是技术视角）
用户打开产品 → 做了什么 → 得到了什么结果

## 非功能需求
- 响应时间：...
- 可用性：...
- 数据安全：...

## 成功指标
怎么判断这个产品做成了？
```

**里程碑验证：** 能用一句话回答"这个产品解决了谁的什么问题"。不能回答，说明 PRD 还不够清晰。

> 📍 工具：`/harness` 查看第1阶段待完成项 · `/harness-next` 通过验证后推进到系统设计阶段

> 产品说清了"做什么"，现在工程师要回答"怎么做"。

三个文档平行输出：

### `docs/SYSTEM_DESIGN.md` — 架构 + 模块

- 系统分层架构图（各层职责）
- 核心模块列表及职责
- 模块之间的数据流路径

### `docs/TECH_SELECTION.md` — 技术选型

每一个选型都要有理由，而不是"我熟悉所以用它"：

```markdown
## 前端框架

|            | Next.js  | Vite + React |
|------------|----------|--------------|
| SSR 支持   | ✅       | ❌           |
| 团队熟悉度 | ✅       | ✅           |
| 部署成本   | 较高     | 低           |
| **结论**   | **选择** | 备选         |

理由：需要 SSR 支持 SEO，且团队熟悉 Next.js 生态。
```

> ⚠️ **`TECH_SELECTION.md` 直接决定第 4 阶段的 monorepo 包结构。**
> 技术选型没定，就没法建 `packages/frontend`。

### `docs/DATA_MODEL.md` — 概念数据模型

只画实体关系，不建表（物理模型在第 3 阶段细化）：

```
User ──1:N──> Session
Session ──1:N──> Record
Record ──1:1──> Evaluation
```

**里程碑验证：** 能画出系统架构图，能回答"系统分几层、每层做什么、用什么技术"。

> 📍 工具：`/harness-check` 确认没有跳过产品阶段 · `/harness-next` 推进到详细设计阶段

> 从抽象走向可执行的技术方案。

### `docs/API_SPEC.md` — 接口规范

**先写接口，再写代码。** 接口是前后端的"合同"，也是 Agent 实现时的精确依据。

```markdown
## 统一响应格式
{ code: number, data: any, message: string }

## 接口列表

### 创建会话
POST /api/v1/sessions
- 请求：{ job_id: string, difficulty: 'easy' | 'medium' | 'hard' }
- 响应：{ session_id: string, questions: Question[] }
- 错误码：40001 参数错误 | 50001 服务内部错误

### 提交答案
POST /api/v1/sessions/:id/answer
- 请求：{ question_id: string, answer: string }
- 响应：{ score: number, feedback: string }
```

### `docs/TEST_STRATEGY.md` — 测试策略

**先定义怎么测，Agent 写代码时才知道"写对了没有"。**

```markdown
## 测试金字塔
- 单元测试（多）：核心算法、工具函数
- 集成测试（中）：API 接口、数据库操作
- E2E 测试（少）：核心用户流程

## 覆盖率要求
- 核心业务逻辑：> 80%
- 工具函数：> 90%

## CI 触发时机
- 单元测试：每次 push
- 集成测试：每次 PR
- E2E 测试：每次部署前
```

**里程碑验证：** `API_SPEC.md` 完成，可以给"假想的前端同学"看，他能据此独立开始开发，不需要问任何问题。

> 📍 工具：`/harness-check` 运行规则2检查（API_SPEC 有无 SYSTEM_DESIGN 支撑）· `/harness-next` 进入 Harness 构建阶段

> **这才是真正的"环境搭建"。**

这一阶段做三件事，顺序很重要：

### ① 按 TECH_SELECTION.md 建 monorepo 结构

选型文档决定了用什么技术，现在才创建对应的目录：

```
my-project/
├── packages/
│   ├── frontend/       ← Next.js（来自选型文档）
│   ├── backend/        ← Node.js + Hono（来自选型文档）
│   └── shared/         ← 共享类型定义
├── docs/               ← 第 1-3 阶段产出的所有设计文档
├── .github/workflows/  ← CI/CD
├── CLAUDE.md           ← 更新：补充架构约束引用
└── package.json        ← workspaces 配置
```

### ② 把设计文档"编译"成 Agent 可读的 Harness 文档

设计文档是给人读的，Harness 文档是给 Agent 读的。这是一个提炼和转化过程：

| 设计文档（第 2-3 阶段） | Harness 文档（第 4 阶段，放根目录） |
|------------------------|-----------------------------------|
| `docs/SYSTEM_DESIGN.md` | `ARCHITECTURE.md` — 架构约束铁律 |
| 模块设计 | `AGENTS.md` — Agent 行为指引 |
| 架构决策 | `RULES.md` — 禁止事项、开发规范 |
| `docs/API_SPEC.md` | 直接引用，不需要转化 |

**`ARCHITECTURE.md` 示例**（从设计文档提炼的精华，写成约束而非描述）：

```markdown
# ARCHITECTURE.md

## 分层规则（铁律）
Frontend → API Gateway → Service → Repository → Database

- ✅ 只能向前依赖（上层依赖下层）
- ❌ 禁止 Service 直接查数据库（必须通过 Repository 层）
- ❌ 禁止 Frontend 跳过 API Gateway 直接调 Service

## AI 调用规则
所有 LLM 调用集中在 `AIService`，其他模块禁止直接调用 AI API。

## 响应格式（铁律）
所有 API 必须返回：`{ code: number, data: any, message: string }`
禁止在 Controller 中直接返回裸数据。

## 详细设计参考
- 接口规范：见 docs/API_SPEC.md
- 数据模型：见 docs/DATA_MODEL.md
```

### ③ 建立自动验证管线

`.github/workflows/ci.yml`：

```yaml
name: CI
on: [push, pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - run: npm ci
      - run: npm run typecheck     # TypeScript 类型检查
      - run: npm run lint          # 架构约束检查（eslint-plugin-import）
      - run: npm run test          # 跑测试（按 TEST_STRATEGY.md 配置）
```

**里程碑验证：** `git push` 后 CI 自动运行。Agent 违反 `ARCHITECTURE.md` 里的架构规则时，CI 自动报错，Agent 读取报错后自己修。

> 📍 工具：`/harness-check` 运行规则3/4检查（ARCHITECTURE.md 是否有完整的前置设计支撑）· `/harness-next` 进入实现阶段

**你的角色变了：从执行者变为审核者。**

给 Agent 的任务要精确引用 Harness 文档：

> 按照 `ARCHITECTURE.md` 的分层规则，
> 参考 `docs/DATA_MODEL.md` 的实体定义，
> 实现 `docs/API_SPEC.md` 中的 `POST /api/v1/sessions` 接口，
> 包含对应的单元测试（测试策略见 `docs/TEST_STRATEGY.md`）。

**当 Agent 写错时，改 Harness，不改代码：**

```
Agent 犯了同一个错误
    ↓
❌ 人类改代码 → 治标不治本，下次还会错
✅ 人类改环境 → Agent 下次自己写对
   ├── 在 ARCHITECTURE.md 补充说明
   ├── 在 RULES.md 添加禁止项
   ├── 增加测试用例让 Agent 能验证
   └── 完善 CI lint 规则自动 enforce
```

这形成了一个持续优化的循环：**Agent 犯错 → 人改环境 → 环境越来越完善 → Agent 产出质量越来越高。**

## 里程碑总表

| 阶段 | 产出文档 | 里程碑验证标准 |
|------|---------|--------------|
| 开始前 | `CLAUDE.md`（最简版） | git 仓库存在，工具装好 |
| 第 1 阶段 | `docs/PRD.md` | 能一句话说清产品目标 |
| 第 2 阶段 | `SYSTEM_DESIGN.md` + `TECH_SELECTION.md` + `DATA_MODEL.md` | 能画出架构图，技术栈已确定 |
| 第 3 阶段 | `API_SPEC.md` + `TEST_STRATEGY.md` | API 文档可给前端独立开发用 |
| 第 4 阶段 | monorepo 结构 + `ARCHITECTURE.md` + `AGENTS.md` + `RULES.md` + CI | push 后 CI 自动运行 |
| 第 5 阶段 | 每个 Sprint 的 PR | 核心功能可以跑起来 |

## 常见误区

| 误区 | 正确做法 |
|------|---------|
| "先把环境搭好再说" | 开始前只做 git init + 工具安装，15 分钟 |
| "第一天就写 ARCHITECTURE.md" | Harness 在系统设计完成之后才能写 |
| "ARCHITECTURE.md 写架构想法" | 写的是已确定的约束，不是草稿或方向 |
| "Agent 写错了就改代码" | 改 Harness 文档、lint 规则、测试用例 |
| "文档太麻烦，直接写代码" | 设计文档质量 = Agent 产出质量，没有捷径 |
| "monorepo 第一天建好" | 技术选型（第 2 阶段）之后才能建包结构 |
| "感觉哪里不对但说不清" | 运行 `/harness-check`，脚本客观报告偏离项 |
| "不记得现在到哪个阶段了" | 运行 `/harness`，读取 PROGRESS.md 展示状态 |
