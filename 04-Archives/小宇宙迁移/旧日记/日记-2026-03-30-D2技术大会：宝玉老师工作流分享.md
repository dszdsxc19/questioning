# D2 技术大会笔记：宝玉老师工作流分享

## 视频播客处理工作流

宝玉老师分享了完整的内容生产工作流：

```
1. YouTube 视频播客
   ↓
2. Gemini 视频字幕
   ↓
3. 按照要求识别任务 + 分段
   ↓
4. 提取出稿子
   ↓
5. 复制到 Claude，用 skills 写稿
   ↓
6. 做三次，三份稿子取好的点
   ↓
7. 到本地，git 管理
   ↓
8. Merge draft skills 合并稿子
   ↓
9. 使用 skill 生成封面图任务
   ↓
10. 插图任务：选择信息图还是混合图
   ↓
11. 优化：发现 skill 有问题 → 让 AI 发现问题然后修复
```

**核心特点**：
- 三次迭代取优，而非一次生成
- Git 管理版本，可追溯可回滚
- 用 Skill 合并稿子，而非手动
- 封面图和插图也是通过 skill 任务生成
- 闭环：发现问题 → 修复 skill

## Skills 设计理念

### 1. 不要做万能 Skill

> "不去学习就不需要学习了，过程性 AI 产品"

**错误做法**：做一个"万能写作助手"，什么都做
**正确做法**：拆成专职 skill，再编排起来

### 2. 专职 + AI 亲和

每个 Skill 只做一件事，但要做到：
- **专职**：单一职责，边界清晰
- **AI 亲和**：Agent 能理解、能调用、能组合

示例：
- ❌ `/content-creator` （万能）
- ✅ `/draft-article` + `/merge-drafts` + `/generate-cover` （编排）

### 3. 拆开再编排

```
单一万能 Skill
   ↓ 拆解

专职 Skill 1 专职 Skill 2 专职 Skill 3
   ↓           ↓           ↓
     编排组合（工作流）
```

### 4. 为 OPC 收敛技术栈

- **收敛技术栈**：让 Agent 不需要学习
- **凝练 Agent**：专职 skill 各司其职
- **代码演示**：Skill hub 可视化展示

## ToB vs ToC 的思考

### ToB：效率
- 解决已有流程的效率问题
- 目标：让事情更快、更省力

### ToC：创造需求
- 1w 个人里满足 1k 人的需求
- 目标：创造新的使用场景

> "过程性 AI 产品" — 不去学习就不需要学习了

## 个人感悟

> "我靠，把我在做的东西做了"
> "我就是只会做需求的程序员"

**共鸣点**：
- 工作流跟自己实践的很像
- 发现已经被别人做了，但方向是对的
- 反思：自己是"只会做需求的程序员"还是"产品构建者"？
