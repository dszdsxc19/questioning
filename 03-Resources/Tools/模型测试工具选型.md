# 模型测试工具选型

## Summary

这页不是为了收藏更多测试工具，而是为了回答一个更稳定的问题：面对基模、Agent、RAG 和系统压测这几类不同对象时，应该优先选哪类工具。

## 这页解决什么问题

很多团队会把“模型能力评测”“Agent 任务评测”“RAG 效果评测”“系统压测”混在一起讨论，最后既没有统一语言，也没有稳定工具栈。

这页的目标是先把测试对象分层，再把工具放回各自边界里。

## 选用原则

收录到这页的工具，优先满足以下条件：

- 能回答清楚“它主要测试什么”
- 有较明确的适用边界，而不是大而全
- 适合成为长期选型参考，而不是版本敏感的功能清单

## 核心内容

### 先分清四类测试对象

1. 基础模型能力评测：看基模在标准任务上的能力边界
2. Agent 任务评测：看一个调用工具、执行流程的系统是否完成任务
3. RAG 专项评测：看检索、上下文注入和回答质量是否健康
4. 系统压测：看服务在目标流量和故障条件下是否还能稳定运行

如果测试对象没有先分清，后面的工具讨论大概率会失焦。

### 基模测试

#### OpenCompass

- 适合：做基础模型的综合 benchmark、横向比较和任务集评测
- 优势：更像一个系统化评测框架，适合严肃地组织多模型、多数据集、多任务的对比
- 更适合的场景：你要回答“这个基模整体能力处在什么位置”

#### lm-evaluation-harness

- 适合：快速评测基础模型在标准任务集上的表现
- 优势：社区认知度高，任务覆盖广，适合作为通用基准入口
- 更适合的场景：你要回答“我现在想快速、标准化地跑一轮基模能力测试”

### Agent 测试

#### DeepEval

- 适合：评估 Agent、LLM 应用和任务执行结果
- 优势：更靠近“应用级评测”，容易把答案质量、任务完成度、RAG 与 Agent 场景放进同一套实践里
- 更适合的场景：你要回答“这个 AI 应用或 Agent 版本到底有没有变好”

#### Promptfoo

- 适合：做 prompt、模型、配置和测试样例的批量对比
- 优势：实验感强，适合把评测当成回归机制和对比实验
- 更适合的场景：你要回答“换 prompt、换模型、换配置后，效果差异到底在哪里”

### RAG 专属

#### Ragas

- 适合：专门评估 RAG 链路
- 优势：边界清楚，不把 RAG 问题混成泛化的 Agent 评测
- 更适合的场景：你要回答“问题到底出在检索、上下文质量，还是最终回答”

### 压测与混沌测试

#### Gatling

- 适合：需要成熟压测体系、复杂场景建模和较强工程化能力的团队
- 优势：偏传统压测体系，适合严肃性能测试和稳定的企业级流程
- 更适合的场景：你要回答“这个服务在复杂负载模型下的性能边界是什么”

#### k6

- 适合：现代 API 和服务压测
- 优势：上手快，脚本表达直接，更适合今天常见的接口压测和持续集成场景
- 更适合的场景：你要回答“我能不能快速把压测纳入日常开发和上线前验证”

### 一个更实用的选型框架

如果你的主要问题是：

- “基模本身强不强”：优先看 `OpenCompass`、`lm-evaluation-harness`
- “Agent 是否完成任务”：优先看 `DeepEval`、`Promptfoo`
- “RAG 到底哪里坏了”：优先看 `Ragas`
- “服务能不能扛流量”：优先看 `k6` 或 `Gatling`

### 不要混淆的三组边界

- 基模 benchmark 不等于 Agent 评测
- RAG 评测不等于通用压测
- 压测工具回答的是系统负载问题，不直接回答模型质量问题
