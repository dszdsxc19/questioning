# 字符编码与字体

## Summary

这页用来区分字符、字符集、字符编码、码位和字体之间的关系，并解释乱码与“锟斤拷”的成因。

## 综述

计算机只认识 0、1 二进制编码。电脑显示文字，涉及三个重要概念：字符、字符集和字符编码。

字符编码类似一张映射表，根据编码规则把字节序列解析成字符。需要注意的是，**码位和索引不是一个东西**。尤其在 UTF-8 这种变长编码里，“索引 -> 字节序列 -> 字符”并不是线性关系。

## 乱码原因

乱码通常发生在：同一份字节流被不同编码规则解析，映射关系错位。

假设一个场景：

我保存了一份 UTF-8 编码的文档，内容如下：

```text
今天中午吃什么？
```

输出如下：

```text
\xE4\xBB\x8A\xE5\xA4\xA9\xE4\xB8\xAD\xE5\x8D\x88\xE5\x90\x83\xE4\xBB\x80\xE4\xB9\x88\xEF\xBC\x9F
```

如果这份 UTF-8 文档被按 ASCII 或 GBK 的规则打开，输出就会变成乱码。

## 锟斤拷

当 GBK 与 Unicode/UTF-8 连续错配时，就会出现经典的“锟斤拷”。

可以把它理解为：

1. 原始内容按 GBK 保存
2. 别人用 UTF-8 打开，无法正确显示，出现替换字符 `�`
3. 再次保存后，这些替换字符被按 UTF-8 编成 `EF BF BD`
4. 又被别人用 GBK 打开
5. 最后连续字节被解释成“锟斤拷”

这不是某个字符神秘出现，而是编码错配被连续放大的结果。

## UTF-8 变长原理

UTF-8 的字节长度不固定。它通过前几个比特位来标记“这个字符占几个字节”。所以 UTF-8 的解析不是按固定长度切片，而是按编码规则逐个解码。

## Emoji 冷知识

Unicode 只规定 emoji 的含义，不规定它们具体长什么样。决定最终显示外观的，是字体。

## 字体与字符编码的关系

字符是抽象概念。字符编码解决“如何存储”，字体文件解决“如何显示”。

一个字体文件中通常包含一个或多个字符映射表（Charmap），它的作用是把字符编码映射到字形索引，再由字形索引找到具体字形进行渲染。

一般渲染步骤：

1. 加载字体文件
2. 确定字体大小
3. 输入字符编码值
4. 通过映射表转换成字形索引
5. 根据索引加载字形
6. 将字形渲染成位图

如果字符编码与字体映射表不匹配，输出的字形要么错误，要么找不到，于是屏幕上会显示方块字。
