# Agent 编排与 Sub-agent

## Summary

这页说明 Agent 编排的核心问题：主代理如何在不污染主上下文的前提下，委托子任务、压缩结果并保持整体控制力。

## 这页解决什么问题

记录这页，不是为了追逐“多 Agent”概念，而是为了回答三个稳定问题：

- 什么情况下应该拆成主代理与子代理
- Sub-agent 真正解决的是哪类工程问题
- 如何避免把“并行”误解成“更高级”

## 选用原则

收录到这页的内容，优先满足以下条件：

- 能帮助判断是否值得引入 Sub-agent
- 能解释 orchestrator-worker 的边界，而不只是给出表面定义
- 能连接到上下文治理、工具调用、结果压缩等更底层问题

## 核心内容

### 一句话定义

Sub-agent 不是多开几个完整助手一起聊天，而是主代理把边界清晰的子任务委托给隔离上下文中的专用执行者。

### 它解决的核心问题

- 上下文污染：主任务、分支探索、工具输出混在一起后，推理质量会下降
- 角色冲突：规划、执行、验证、整理等任务混在一个代理里，容易互相干扰
- 协调失衡：主代理如果沉到局部细节，整体调度能力会下降

### 基本模型

- 主代理负责：理解目标、拆任务、决定是否委托、汇总结果、对外输出
- 子代理负责：在隔离上下文中完成单一任务，并返回压缩结果

这更接近 `orchestrator-worker`，而不是去中心化多代理网络。

### 典型工作流

1. 主代理判断某个子任务是否值得拆分
2. 为子代理创建新上下文，只注入必要任务说明和权限
3. 子代理独立探索、调用工具、产出结果
4. 主代理只接收压缩结论，而不是完整过程

### 委托时真正传递的是什么

在工程实现上，委托通常至少包含两类信息：

- `description`：给人看的任务摘要，帮助主代理和日志理解这次委托在做什么
- `prompt`：真正交给子代理执行的详细说明，定义边界、目标和输出要求

关键点不是“把主会话全部复制过去”，而是只带上足以完成当前子任务的最小上下文。

### 什么时候值得用

- 需要并行探索多个方向
- 某段分析很重，可能污染主上下文
- 子任务需要不同角色、不同技能或不同工具权限
- 返回值可以被压缩成结论、摘要或引用，而不必带回全部过程

### 什么时候不值得用

- 子任务边界并不清晰
- 主代理仍然需要看到全部上下文细节
- 委托成本高于上下文污染成本
- 只是为了“显得复杂”而拆分

### 与真正多 Agent 系统的区别

- 不是平级自治网络，而是严格层级调度
- 没有默认共享记忆
- 子代理通常不直接相互通信
- 主代理仍是唯一对外负责的协调中心

### 设计上的真正难点

- 不是“能不能调起子代理”，而是任务边界划得够不够清
- 不是“并行越多越好”，而是上下文隔离收益是否大于调度成本
- 不是“子代理能做多少”，而是主代理最后还能不能稳定汇总和裁决
