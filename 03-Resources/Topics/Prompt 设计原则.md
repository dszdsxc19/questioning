# Prompt 设计原则

## Summary

Prompt 的价值不在于堆砌措辞，而在于把任务压成可复用、可验证、可审计的稳定接口。

## 这页解决什么问题

这页用来回答两个问题：

- 什么样的 prompt 值得进入长期知识库
- 如何区分“能稳定复用的 prompt 资产”和“只会制造信息糖的 prompt 收藏”

## 选用原则

只有满足以下条件的 prompt 内容，才值得保留：

- 能跨多次任务复用并节省时间
- 能抽象出结构、判断标准或迭代方法
- 不强依赖某个短期平台、流行风格或一次性素材

## 核心内容

### 一句话定义

Prompt 不是给模型“说清楚”，而是把角色、上下文、目标、约束和输出协议压成稳定接口。

### 稳定 Prompt 的组成

- Role：模型扮演谁
- Context：它在什么背景和限制下工作
- Objective：任务到底要完成什么
- Constraints：哪些不能做，哪些必须遵守
- Output：输出格式要多严格

### 常见高价值用途

- 评审型：让模型扮演挑剔专家做追问或挑错
- 编排型：把复杂工程任务变成可执行流程
- 分类型：意图识别、质检、审核
- 抽取型：把非结构化输入稳定转成结构化输出

像质检和意图识别这类任务，真正的关键不是“描述得像不像人”，而是把规则写成明确命中条件：

- 明确只看哪部分输入
- 给出否定处理规则
- 避免上下文污染
- 把输出格式锁死，尤其是 JSON

如果这些边界不先写清楚，模型很容易把“像是”误判成“就是”。

### 写成可复用资产时应该保留什么

- 稳定模板：未来可以直接复制修改的 prompt
- 词汇库：帮助你更精确描述风格、动作、约束的词
- 结构拆解：解释一个好 prompt 为什么有效
- 适用边界：什么场景适合，什么场景不适合

### 判断一条 Prompt 是否值得保留

保留：

- 它能在未来反复节省你的决策时间
- 它自带明确命中条件、约束和输出协议
- 它能训练你的判断力，而不是只提供一段文案

不保留：

- 只是一时流行的风格 prompt
- 严重依赖某个平台当前行为
- 只是“抽卡试试看”，没有总结出可迁移结构

### Prompt 迭代的正确目标

不是追求一次写出完美版本，而是先识别什么样的输出算“好”，再围绕好的样本做收敛和约束。

如果一条 prompt 完全靠不断“抽卡”偶尔命中好结果，却没有把好结果背后的结构、约束和词汇提炼出来，那它还不算资产。

真正可复用的迭代方式更接近：

1. 先识别什么输出值得保留
2. 回看它为什么好
3. 把有效部分抽成结构、词汇或约束
4. 再继续收敛，而不是每次从头碰运气

### 什么时候该升级成 Skill

当一组 prompt 已经不只是措辞，而是带有稳定 workflow、脚本、模板和触发逻辑时，它更接近 skill，而不是普通模板。

### 面试训练和工程编排为什么是高杠杆场景

这两类 prompt 都说明了一件事：高价值 prompt 往往不是为了“生成一段好文案”，而是为了稳定模拟一种工作角色。

例如：

- 面试训练：让模型扮演挑剔面试官，限定问题范围，并在回答后继续追问
- 工程编排：提前声明库选型、复用边界、禁止重写项和模块拆分要求

这类 prompt 的价值，在于它把角色、约束和执行顺序一起压成了接口。

### 写 prompt 时人真正负责什么

AI 可以帮你批量生成候选版本，但人至少要负责两件事：

- 判断什么结果算好
- 判断这条 prompt 是否真的值得长期保存

这也是为什么“提示词收藏夹”很容易变成信息糖，而 prompt 资产很少。
