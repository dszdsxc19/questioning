# 性能压测与容量规划

## Summary

性能压测的意义，不是“打一下接口”，而是建立对系统容量、退化点和高可用边界的真实认知。

## 这页解决什么问题

很多团队会把压测、高可用、扩容、故障恢复和上线风险混成一团，最后既没有容量结论，也没有明确负责人。

这页用来建立一个更稳定的理解框架：

- 压测到底在回答什么问题
- 它和高可用是什么关系
- 容量应该如何估
- 压测应该如何分层做

## 选用原则

收录到这页的内容，优先满足以下条件：

- 能帮助判断容量与瓶颈，而不只是罗列指标
- 能连接压测、观测和高可用设计
- 能服务真实项目的上线判断

## 核心内容

### 压测真正回答什么问题

性能压测至少要回答五件事：

1. 当前系统在目标流量下是否稳定
2. 瓶颈在哪里
3. 流量继续上升时，先退化的是哪一层
4. 长时间运行是否会暴露连接、内存、慢查询或上游超时问题
5. 版本变更后是否发生性能回退

### 常见测试类型

- `Smoke Test`：先确认链路可用，避免一上来把环境打坏
- `Load Test`：验证目标流量下是否满足 SLA / SLO
- `Stress Test`：持续加压，找到开始退化的拐点
- `Spike Test`：模拟突发流量
- `Soak Test`：长时间施压，观察泄漏与资源膨胀

### 为什么不能只看平均响应时间

平均值会掩盖尾延迟。

至少要同时看：

- `p50`
- `p95`
- `p99`
- 错误率
- 超时率
- 饱和度

如果平均值好看但 `p95/p99` 很差，线上用户仍然会感知明显卡顿。

### Open Workload 比 Closed Workload 更适合理解 API 服务

- `Open Workload`：请求按固定速率到达，例如 `75 QPS`
- `Closed Workload`：固定并发用户循环发请求，系统越慢，实际吞吐越低

对于 API 服务，尤其是被前端、任务系统或其他服务调用的接口，优先按 `QPS/RPS` 建模通常更贴近真实世界。

### 压测和高可用不是一回事

压测关注的是：

- 吞吐
- 延迟
- 瓶颈
- 退化点

高可用关注的是：

- 单点故障后是否仍可服务
- 依赖异常时是否能降级、隔离、重试、熔断
- 多实例、多节点下是否还能持续服务
- 故障恢复时间与恢复手段

更直接地说：

- 压测在问“系统多快、多稳、能扛多大流量”
- 高可用在问“出故障时系统还能不能继续活着”

但压测通常是高可用建设的重要输入，因为没有容量数据支撑的高可用设计，往往只是口号。

### 谁该主导这件事

- 后端主导：链路理解、场景设计、瓶颈定位
- 测试 / QA / SRE 协同：压测执行、环境协调、监控与平台接入
- 后端与 SRE 共同负责：高可用架构设计

一句话概括：

> 压测不是纯测试工作，也不是只属于运维；对后端服务来说，后端必须主导。

### 容量估算先看在途请求数

对于依赖 SSO、数据库、外部模型或其他远程服务的系统，容量上限不由应用实例单独决定，而由整条调用链共同决定。

最基础的公式是：

```text
并发数 = QPS × 平均耗时（秒）
```

例如：

- `75 QPS`
- 平均耗时 `2s`

则仅这一层就会形成大约 `150` 个在途请求。

如果再叠加重试、数据库写入和上游远程依赖，系统压力会明显大于表面上的入口 QPS。

### 一个更实用的容量估算步骤

1. 先拆单请求成本：鉴权、读写库、外部模型、缓存、重试
2. 再做稳态估算：目标流量下每一层每秒要承受多少请求或写入
3. 再换算在途并发：看哪些层最容易先被打满
4. 再考虑故障放大：重试、fallback、超时重发会如何放大压力

### 压测应该分层做

不要一上来就做全链路重压。

更稳妥的顺序是：

1. 基线压测：先看应用自身、路由、存储与数据库上限
2. 半链路压测：保留一类真实依赖，隔离另一类依赖，定位瓶颈
3. 全链路压测：得到更接近生产的真实容量数据
4. 长稳压测：验证 30 分钟、1 小时甚至更长时间是否出现资源泄漏或性能退化

### 合格的压测流程至少包含什么

#### 第一步：定义目标

- 目标接口
- 目标流量
- 目标时长
- 通过标准
- 关注指标

#### 第二步：准备请求样本

不要只测最短请求，至少准备小、中、大三档样本。

#### 第三步：定义施压模型

推荐采用按固定到达率逐档施压，例如：

- `10 QPS`
- `25 QPS`
- `50 QPS`
- `75 QPS`
- `100 QPS`

#### 第四步：建立观测面

压测不能只看压测工具输出，还要同时观察：

- 应用日志
- CPU / 内存
- 事件循环延迟
- DB 连接数与慢查询
- 上游接口耗时、429、超时和错误率

#### 第五步：输出结论

至少回答：

1. 哪个流量档位稳定
2. 哪个流量档位开始退化
3. 首个瓶颈点是什么
4. 是否有重试放大
5. 下一步优化建议

### 优化优先级应该如何判断

如果目标是把一条依赖链复杂的接口稳定推到更高流量，通常优先考虑：

1. 入口限流与并发保护
2. 外部依赖打点与可观测性
3. 鉴权或高频远程调用的短 TTL 缓存
4. 明确区分基线压测与全链路压测
5. 必要时提供降级、快速失败或 fallback
