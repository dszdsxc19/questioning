## 一句话结论

当操作符两边数据类型不一致时，MySQL 会发生隐式转换。

- `where` 左边字段是**数值类型**，右边传字符串时，通常是把**常量值**转成数字，索引一般还能用，性能影响不大，但不推荐。
- `where` 左边字段是**字符类型**，右边传数字时，通常是把**字段列**转成数字再比较，会导致**索引失效**，退化成**全表扫描**。

> 面试里直接记一句：**转换发生在常量上，问题小；转换发生在字段上，索引基本就废了。**

---

## 为什么会这样？

索引能工作的前提，是 MySQL 可以拿查询值直接去 B+ 树里定位。

- 如果转换的是右边常量，比如 `int_col = '123'`
  常量 `'123'` 先转成数字 `123`，再去查索引，问题不大。
- 如果转换的是左边字段，比如 `varchar_col = 123`
  每一行的 `varchar_col` 都要先转成数字，再逐行比较，索引的有序性就用不上了。

本质上这和对索引列做函数、计算是一个问题：

```sql
WHERE CAST(varchar_col AS UNSIGNED) = 123
```

这种写法本来就很烂，隐式转换只是把这件烂事偷偷替你做了。

---

## 字符串转数字的规则

字符串转成数值时，要记住这两条：

1. 不是数字开头，结果就是 `0`
2. 是数字开头，就截取到第一个非数字字符为止

例子：

```text
'abc'      -> 0
'abc123'   -> 0
'123abc'   -> 123
'10000a'   -> 10000
'010000'   -> 10000
```

这就是为什么 `varchar` 字段和数字比较会出大问题。

因为下面这些不同字符串，转换后可能是同一个值：

```text
'10000'    -> 10000
'10000a'   -> 10000
'010000'   -> 10000
```

既然多条字符串都能映射到同一个数字，MySQL 就没法通过原始字符串索引直接精准定位，只能扫表。

---

## 面试回答模板

如果面试官问“SQL 隐式转换为什么会导致索引失效”，可以直接这么答：

> MySQL 在比较操作符两边类型不一致时会做隐式转换。  
> 如果转换发生在常量值上，比如 `int_col = '123'`，索引通常还能用。  
> 但如果转换发生在字段列上，比如 `varchar_col = 123`，MySQL 需要把每一行字段值先转成数字再比较，本质上等于对索引列做函数计算，会破坏索引可用性，导致全表扫描。  
> 另外字符串转数字不是一一映射，像 `'10000'`、`'10000a'`、`'010000'` 都可能转成 `10000`，这也是索引无法精准命中的根本原因。

---

## 实践建议

- 查询参数类型必须和字段类型保持一致
- 不要拿数字去查 `varchar`
- 不要对索引列做函数、运算、显式/隐式类型转换
- 如果字段表达的是数字语义，建表时就该用数字类型
