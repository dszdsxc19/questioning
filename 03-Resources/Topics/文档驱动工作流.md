# 文档驱动工作流

## Summary

在 AI 协作里，文档的价值不只是记录信息，而是作为执行边界、约束系统和协作接口，直接决定后续产出质量。

## 这页解决什么问题

这页用来回答一个稳定问题：

- 为什么在 Agent 参与执行后，文档会从“事后记录”变成“事前约束”

## 选用原则

收录到这页的内容，优先满足以下条件：

- 能解释文档如何影响执行质量
- 能帮助区分“值得写的文档”和“纯速查信息”
- 能连接到约束、工作流和协作边界

## 核心内容

### 一句话定义

文档驱动工作流不是“多写文档”，而是把目标、边界、约束和接口先写清楚，再让人或 Agent 按这个结构执行。

### 为什么 AI 时代文档角色变了

当执行成本大幅下降后，真正稀缺的不再是“把东西做出来”，而是：

- 问题定义
- 边界约束
- 成功标准
- 协作接口

这几件事如果不先写清楚，后续只会更快地产生更多偏差。

### 什么文档值得写

值得写的是会持续影响执行质量的文档，例如：

- 目标和范围说明
- 约束与非目标
- 数据和接口边界
- 评审标准
- 交付标准

不值得长期维护的，通常是：

- 一次性的安装步骤
- 随时可查官方文档的菜单与参数
- 只为了“以后也许会用到”的速查抄录

### 文档不是记录，而是执行接口

在 Agent 协作里，文档更像：

- 规格说明
- 权限边界
- 上下文压缩包
- 团队之间的稳定接口

如果文档只是事后总结，它对执行阶段的帮助会很弱。  
如果文档先于执行，它就能直接约束后续行为。

### 文档驱动不等于文档中心主义

重点不是页面更多，而是把真正高杠杆的信息写清楚。

所以文档驱动工作流的核心判断是：

- 这份文档是否能减少返工
- 是否能降低误解
- 是否能让后续执行更稳定

如果不能，默认不写。
