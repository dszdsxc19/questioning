# 扫码登录场景题

> [!tip] 方法论参考
> 本题涉及认证系统设计，可参考 [[指导思想/从需求到上线——完整的系统设计方法论]] 中的系统架构设计和依赖系统分析。

## 题目描述

**请详细描述扫码登录的实现原理，包括同一产品内的扫码登录和第三方应用的扫码登录两种场景。**

## 核心原理

扫码登录本质上是一种**跨设备的凭证传递机制**。它利用移动端已经建立的安全会话状态，为桌面端或网页端授权并建立全新的会话。

底层逻辑包含：**状态轮询、二维码状态机转移、安全令牌下发**。

## 场景一：同一产品中的扫码登录

### 适用场景
- 桌面端软件登录（如微信PC版、钉钉桌面版）
- 自家产品的网页端
- **特点**：系统内部互相信任，流程相对直接

### 完整技术流程

```
┌─────────────┐         ┌─────────────┐         ┌─────────────┐
│   桌面端    │    ↔    │   服务端    │    ↔    │   移动端    │
└─────────────┘         └─────────────┘         └─────────────┘
```

#### Step 1: 桌面端初始化请求
```
用户点击"扫码登录"
    ↓
桌面端向服务端发起请求
    ↓
服务端生成全局唯一 UUID
    ↓
Redis 缓存 UUID 状态 = "待扫描" (WAITING_SCAN)
```

#### Step 2: 生成并展示二维码
```
桌面端获取 UUID
    ↓
拼接时间戳/业务参数生成二维码
    ↓
展示给用户
    ↓
建立长连接(WebSocket) 或 长轮询 查询状态
```

#### Step 3: 移动端扫码与解析
```
用户打开已登录的 App
    ↓
扫描屏幕二维码
    ↓
App 解析出 UUID
```

#### Step 4: 状态更新（已扫码）
```
手机 App 发送请求: UUID + 移动端设备信息
    ↓
服务端接收并验证
    ↓
UUID 状态: "待扫描" → "已扫码" (SCANNED)
    ↓
桌面端监听到状态变化
    ↓
界面更新: "扫描成功，请在手机上确认"
```

#### Step 5: 移动端确认授权
```
用户在 App 上点击"确认登录"
    ↓
手机 App 提交: UUID + User Token (当前登录凭证)
```

#### Step 6: 服务端验证与会话创建
```
服务端验证 User Token 有效性
    ↓
确认用户身份
    ↓
为桌面设备生成新凭证: New Token / Session ID
    ↓
绑定 UUID 与 新凭证
    ↓
UUID 状态: "已确认" (CONFIRMED)
```

#### Step 7: 桌面端登录成功
```
桌面端监听到"已确认"状态
    ↓
获取服务端下发的新凭证
    ↓
写入本地存储
    ↓
登录完成 ✓
```

## 场景二：第三方应用的扫码登录

### 适用场景
- 使用微信扫码登录第三方网站
- 使用支付宝扫码登录其他应用
- **特点**：基于 OAuth 2.0 / OIDC 协议，安全与授权边界发生变化

### 核心角色（OAuth 2.0）
| 角色 | 说明 |
|------|------|
| **资源所有者** | 用户 |
| **客户端** | 第三方应用 |
| **授权服务器** | 平台方（微信/支付宝等） |
| **资源服务器** | 存储用户数据的服务器 |

### 完整技术流程

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│ 第三方网页  │ →  │ 平台方服务  │ ←  │ 平台方 App  │ ←  │    用户     │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

#### Step 1: 第三方请求授权二维码
```
第三方网页端向平台方开放接口发起请求
    ↓
平台方生成二维码内容:
  - 第三方 AppID
  - 授权作用域 (Scope)
  - 唯一 UUID
    ↓
返回给网页端展示
    ↓
网页端开始监听 UUID 状态
```

#### Step 2: 平台方 App 扫码触发授权
```
用户使用平台方 App 扫描二维码
    ↓
App 解析出: 第三方 AppID
    ↓
向服务端拉取第三方应用信息 (名称、Logo)
```

#### Step 3: 移动端展示授权页面并确认
```
App 弹出授权提示界面:
  "是否允许 [第三方应用] 获取你的基础信息?"
    ↓
用户点击"同意"
```

#### Step 4: 下发临时授权码（Auth Code）
```
手机 App 提交授权指令
    ↓
平台方服务端验证通过
    ↓
生成临时授权码 (Authorization Code)
    ↓
更新 UUID 状态
    ↓
返回 Auth Code (生命周期极短，如 10 分钟)
```

#### Step 5: 第三方网页端换取访问令牌
```
第三方网页端监听到扫码完成
    ↓
获取 Auth Code
    ↓
第三方后端发起服务器级请求:
  POST /oauth/token {
    code: AuthCode,
    app_id: AppID,
    app_secret: AppSecret
  }
    ↓
平台方验证并返回:
  - access_token
  - refresh_token
  - expires_in
```

#### Step 6: 获取用户信息与完成登录
```
第三方后端携带 access_token
    ↓
调用平台方用户信息接口
    ↓
获取用户公开标识 (OpenID / UnionID)
    ↓
第三方数据库查找该标识:
  - 存在 → 直接建立登录态
  - 不存在 → 静默注册新账号并登录
    ↓
登录完成 ✓
```

## 核心差异对比

| 对比维度 | 同一产品 | 第三方应用 |
|---------|---------------------|---------------------|
| **信任机制** | 系统内部绝对信任 | 基于 OAuth 2.0 建立跨域信任 |
| **凭证传递** | 直接派生登录态 | 通过中间人机制交换 |
| **Token 下发** | 服务端直接下发终态 Token | 授权码 → 换取 Token（后端握手） |
| **安全层级** | 相对简单 | 多一层授权码交换防暴露 |
| **协议标准** | 自定义协议 | OAuth 2.0 / OIDC |

## 二维码状态机

```
┌──────────────┐
│  待扫描      │  (WAITING_SCAN)
│  (初始状态)  │
└──────┬───────┘
       │ 用户扫码
       ↓
┌──────────────┐
│  已扫码      │  (SCANNED)
└──────┬───────┘
       │ 用户确认
       ↓
┌──────────────┐
│  已确认      │  (CONFIRMED)
└──────┬───────┘
       │
       ↓
┌──────────────┐
│  已取消/过期  │  (CANCELLED / EXPIRED)
└──────────────┘
```

## 技术要点总结

### 前端（桌面端/网页端）
- **长连接**: WebSocket 实时监听状态变化
- **长轮询**: 定时（如 2s）查询二维码状态
- **二维码展示**: qrcode.js 或类似库生成

### 后端
- **唯一标识**: UUID / 短链 ID
- **状态存储**: Redis (设置 TTL 过期时间)
- **凭证管理**: JWT / Session ID
- **OAuth 2.0**: PKCE、state 参数防 CSRF

### 安全措施
1. **动态二维码**: 短时有效（如 5 分钟）、唯一加密
2. **双向验证**: 设备指纹 + 生物特征
3. **HTTPS 全程加密**
4. **智能风控**: 异常检测、地理位置分析

## 扩展问题

1. **如何防止二维码被截图后恶意使用？**
   - 二维码绑定设备指纹
   - 设置极短有效期（30s-60s）
   - 扫码后限时确认

2. **长轮询 vs WebSocket 如何选择？**
   - 长轮询: 实现简单，兼容性好，服务端有压力
   - WebSocket: 实时性更好，节省资源，实现复杂

3. **为什么第三方应用需要授权码而不是直接下发 Token？**
   - 防止 Access Token 在浏览器前端暴露
   - 通过后端服务器交换更安全
   - 授权码只能使用一次，用完即失效
