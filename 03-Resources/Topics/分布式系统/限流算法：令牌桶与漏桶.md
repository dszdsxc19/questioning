## 一句话总结

**令牌桶** 和 **漏桶** 是两种经典的限流算法。令牌桶允许一定程度的突发流量，更适合互联网场景；漏桶强行平滑流量，适合网络流量整形。

---

## 核心对比

| 特性 | 令牌桶 (Token Bucket) | 漏桶 (Leaky Bucket) |
|-----|----------------------|---------------------|
| **桶的用途** | 存储令牌 | 存储请求 |
| **流量处理** | 允许突发流量 | 强行平滑流量 |
| **空桶时** | 请求被拒绝 | 请求排队等待 |
| **满桶时** | 允许瞬间大量请求通过 | 按固定速率处理 |
| **适用场景** | API 限流、网关（推荐） | 网络流量整形、消息队列 |

---

## 令牌桶算法 (Token Bucket)

### 工作原理

```
        ┌─────────────────────────────────────┐
        │         令牌桶（容量为 N）            │
        │                                     │
        │   [Token] [Token] [Token] ...       │
        │                                     │
        └──────────────┬──────────────────────┘
                       │
        ┌──────────────┴──────────────────────┐
        │     恒定速率 r 放入令牌（每秒）        │
        └─────────────────────────────────────┘

        请求到达 → 获取令牌 → 成功/拒绝
```

### 场景案例：电商秒杀

> **核心价值**：令牌桶利用前期积攒的令牌，在保护系统的同时最大化瞬间吞吐量。

**参数设定**
- 桶容量 $C$：100 个令牌
- 令牌生成速率 $R$：每秒 10 个令牌

**阶段一：蓄水期（活动开始前）**
- 9:59:50，系统冷清，几乎无请求
- 网关按 10/秒 速率生成令牌
- 9:59:59，桶里积攒了 90 个令牌
- 10:00:00，桶满（100 个令牌），多余令牌被丢弃

**阶段二：突发高并发（秒杀瞬间）**
- 10:00:00，瞬间 150 个请求打到网关
- 前 100 个请求幸运拿走全部令牌，成功打入后端
- 剩余 50 个请求令牌获取失败，返回 `429 Too Many Requests`
- → 体现了令牌桶"允许突发流量"的核心特性

**阶段三：常态化限流（活动持续中）**
- 10:00:01，网关生成 10 个新令牌
- 若有 30 个新请求到达，仅 10 个能通过，20 个被拦截
- 系统最高处理速度被平滑限制在 10/秒

**对比漏桶**
- 漏桶会把 150 个请求排队，严格按 10/秒 处理
- 虽然绝对平滑，但用户体验极差，大多数人会干等
- 令牌桶用积攒的 100 个令牌秒级处理 100 个请求，兼顾保护与吞吐

### 机制说明

| 元素 | 含义 |
|-----|------|
| **桶容量 N** | 最多能存储 N 个令牌，满了就丢弃 |
| **放入速率 r** | 以恒定速率 r（令牌/秒）放入令牌 |
| **请求处理** | 每个请求消耗 1 个令牌 |

### 工作流程图

```
时间轴：─────────────────────────────────────→

0s    1s    2s    3s    4s    5s
 │     │     │     │     │     │
 ▼     ▼     ▼     ▼     ▼     ▼
+1    +1    +1    +1    +1    +1    ← 恒定放入令牌

桶状态（容量 N=5，速率 r=1）：

0s: [1]     ← 1个令牌
1s: [2]     ← 2个令牌
2s: [3]     ← 3个令牌
3s: [4]     ← 4个令牌
4s: [5]     ← 5个令牌（满）
5s: [5]     ← 仍5个（多余丢弃）

突发流量处理：
   4.5s 时桶满，瞬间来 5 个请求 → 全部通过！
```

### 代码实现

#### Go 版本

```go
type TokenBucket struct {
    capacity int           // 桶容量
    rate     int           // 放入速率（令牌/秒）
    tokens   int           // 当前令牌数
    lastTime time.Time     // 上次放入时间
    mu       sync.Mutex    // 保证并发安全
}

func NewTokenBucket(capacity, rate int) *TokenBucket {
    return &TokenBucket{
        capacity: capacity,
        rate:     rate,
        tokens:   capacity,  // 初始满桶
        lastTime: time.Now(),
    }
}

func (tb *TokenBucket) Allow() bool {
    tb.mu.Lock()
    defer tb.mu.Unlock()

    now := time.Now()
    elapsed := now.Sub(tb.lastTime).Seconds()

    // 计算应该放入的令牌数
    tokensToAdd := int(elapsed * float64(tb.rate))
    tb.tokens = min(tb.capacity, tb.tokens + tokensToAdd)
    tb.lastTime = now

    // 尝试获取令牌
    if tb.tokens > 0 {
        tb.tokens--
        return true  // 允许通过
    }
    return false  // 限流
}

func min(a, b int) int {
    if a < b { return a }
    return b
}
```

#### JavaScript 版本

```javascript
class TokenBucket {
    constructor(capacity, rate) {
        this.capacity = capacity;      // 桶容量
        this.rate = rate;              // 速率（令牌/秒）
        this.tokens = capacity;        // 当前令牌数
        this.lastTime = Date.now();    // 上次补充时间
    }

    allow() {
        const now = Date.now();
        const elapsed = (now - this.lastTime) / 1000;  // 转换为秒

        // 补充令牌
        const tokensToAdd = Math.floor(elapsed * this.rate);
        this.tokens = Math.min(this.capacity, this.tokens + tokensToAdd);
        this.lastTime = now;

        // 尝试获取令牌
        if (this.tokens > 0) {
            this.tokens--;
            return true;  // 允许
        }
        return false;  // 拒绝
    }
}

// 使用示例
const bucket = new TokenBucket(10, 2);  // 容量10，每秒2个

// 模拟请求
setInterval(() => {
    const allowed = bucket.allow();
    console.log(`${new Date().toISOString()}: ${allowed ? '通过' : '限流'}`);
}, 100);
```

#### Redis + Lua（分布式）

```lua
-- token_bucket.lua
local key = KEYS[1]                 -- 令牌桶 key
local capacity = tonumber(ARGV[1])  -- 容量
local rate = tonumber(ARGV[2])      -- 速率（令牌/秒）
local now = tonumber(ARGV[3])       -- 当前时间戳（秒）

local bucket = redis.call('HMGET', key, 'tokens', 'last_time')
local tokens = tonumber(bucket[1]) or capacity
local last_time = tonumber(bucket[2]) or now

-- 计算需要补充的令牌
local elapsed = math.max(0, now - last_time)
local new_tokens = math.min(capacity, tokens + elapsed * rate)

-- 判断是否有令牌
if new_tokens >= 1 then
    redis.call('HMSET', key, 'tokens', new_tokens - 1, 'last_time', now)
    redis.call('EXPIRE', key, 600)
    return 1  -- 允许
else
    redis.call('HMSET', key, 'tokens', new_tokens, 'last_time', now)
    return 0  -- 拒绝
end
```

## 漏桶算法 (Leaky Bucket)

### 工作原理

```
        请求请求请求请求
            │    │    │
            ▼    ▼    ▼
        ┌─────────────────┐
        │    漏桶队列     │  ← 可以短暂积压
        │   (容量 N)      │
        └────────┬────────┘
                 │
                 ▼ 固定速率 r 漏出
            ┌─────────┐
            │ 处理请求 │
            └─────────┘
```

### 机制说明

| 元素 | 含义 |
|-----|------|
| **桶容量 N** | 最多缓存 N 个请求 |
| **漏出速率 r** | 恒定速率 r（请求/秒）处理请求 |
| **请求处理** | 超出容量的请求直接丢弃 |

### 流量图示

```
输入流量（突发）：
│ │    │ │ │ │      │ │ │ │ │ │ │
│   │ │    │     │ │    │           │

输出流量（平滑）：
│   │   │   │   │   │   │   │   │   │  ← 恒定速率，无突发
```

### 代码实现

#### JavaScript 版本

```javascript
class LeakyBucket {
    constructor(capacity, rate) {
        this.capacity = capacity;      // 桶容量
        this.rate = rate;              // 漏出速率（请求/秒）
        this.queue = [];               // 请求队列
        this.lastLeakTime = Date.now();
    }

    allow() {
        this._leak();

        if (this.queue.length < this.capacity) {
            this.queue.push(Date.now());
            return true;  // 允许加入队列
        }
        return false;  // 队列满，拒绝
    }

    _leak() {
        const now = Date.now();
        const elapsed = (now - this.lastLeakTime) / 1000;
        const leakCount = Math.floor(elapsed * this.rate);

        // 从队列中移除已处理的请求
        this.queue.splice(0, leakCount);
        this.lastLeakTime = now;
    }
}
```

## 两种算法对比详解

### 流量处理方式

```
场景：桶容量=5，速率=1/秒

令牌桶：
│ │        │ │ │ │      │ │ │ │ │ │ │  ← 允许突发
│   │ │    │     │ │    │           │
（桶满时瞬间可处理 5 个请求）

漏桶：
│   │   │   │   │   │   │   │   │   │  ← 强行平滑
（无论输入多快，输出永远恒定）
```

### 适用场景

| 场景 | 推荐算法 | 理由 |
|-----|---------|------|
| **API 网关限流** | 令牌桶 | 允许应对突发流量，用户体验更好 |
| **防爬虫** | 令牌桶 | 正常用户够用，爬虫被限 |
| **数据库保护** | 令牌桶 | 短时突发可控 |
| **网络流量整形** | 漏桶 | 强行平滑，保护网络 |
| **消息队列消费** | 漏桶 | 恒定速率处理，平滑消费 |

### 实际参数配置示例

```javascript
// API 限流：允许一定突发
const apiLimiter = new TokenBucket(100, 10);
// 含义：平时10/s，桶满时允许瞬间100个

// 严格限流：不允许突发
const strictLimiter = new LeakyBucket(10, 10);
// 含义：严格10/s，多余的排队或丢弃

// 防刷：正常用户够用
const antiSpam = new TokenBucket(5, 1);
// 含义：正常用户1秒1次够用，爬虫被限
```

## 业界实践

### 1. Nginx 限流

```nginx
# 令牌桶实现
limit_req_zone $binary_remote_addr zone=api:10m rate=10r/s;

location /api {
    limit_req zone=api burst=20 nodelay;
    # burst=20: 允许突发20个请求
    # nodelay: 不延迟处理，直接通过/拒绝
}
```

### 2. Sentinel（阿里巴巴）

```java
// 令牌桶模式
@SentinelResource(value = "test", blockHandler = "handle")
public Result test() {
    return doSomething();
}

// 配置：QPS = 100，允许突发
```

### 3. Guava RateLimiter

```java
// 令牌桶实现
RateLimiter limiter = RateLimiter.create(10.0);  // 10 tokens/second

// 允许突发：阻塞获取
limiter.acquire();  // 阻塞直到获取到令牌

// 非阻塞：立即返回
if (limiter.tryAcquire()) {
    // 获取成功，处理请求
}
```

## 优缺点总结

### 令牌桶

| 优点 | 缺点 |
|-----|------|
| ✅ 允许应对突发流量 | ❌ 突发可能持续打满下游 |
| ✅ 实现简单，参数直观 | ❌ 需要合理设置容量 |
| ✅ 可分布式实现 | |

### 漏桶

| 优点 | 缺点 |
|-----|------|
| ✅ 强行平滑流量 | ❌ 无法应对突发 |
| ✅ 保护下游稳定 | ❌ 用户体验可能较差 |
| ✅ 适合流量整形 | |

## 面试回答模板

> 限流是保护高并发系统的重要手段，常用的两种算法是令牌桶和漏桶。
>
> **令牌桶**以恒定速率向桶中放入令牌，请求需要获取令牌才能通过。它的特点是允许一定程度的突发流量——当桶中累积了足够令牌时，可以瞬间处理大量请求。这更适合互联网 API 场景。
>
> **漏桶**则是以恒定速率处理请求，多余请求要么排队要么丢弃。它强行平滑流量，不允许任何突发，更适合网络流量整形。
>
> 在实际项目中，我推荐使用令牌桶算法，因为它在保护系统的同时，给用户更好的体验。分布式场景下可以用 Redis + Lua 实现，配合网关（如 Nginx、API Gateway）进行限流配置。
