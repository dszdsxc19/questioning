# Saga 分布式事务

## Summary

Saga 用补偿事务代替分布式锁和 2PC，把一个大的分布式事务拆成多个本地事务序列，并在失败时逆序执行补偿。

## 为什么需要 Saga

下单流程会跨越多个服务：

1. 订单服务创建订单
2. 库存服务锁定库存
3. 优惠券服务核销优惠券
4. 支付服务发起支付

如果步骤 4 失败，就必须把前面已经成功的动作补回来。

## 核心思想

```text
正向流程：
T1 -> T2 -> T3 -> T4

失败补偿：
C3 -> C2 -> C1
```

补偿不是数据库回滚，而是业务层面的逆向操作。

## 两种模式

### 编排模式（Choreography）

没有中央控制器，服务之间通过发布 / 订阅事件来驱动流程。

优点：

- 服务间松耦合
- 无单点故障
- 扩展灵活

缺点：

- 事件流难追踪
- 补偿逻辑分散
- 调试困难

### 编制模式（Orchestration）

由一个中央 Saga 协调器统一控制整个流程。

优点是流程清晰、易追踪，缺点是协调器会变成复杂核心组件。
