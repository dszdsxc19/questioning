# Agent 评估

## Summary

Agent 评估的重点不只是最终答案对不对，而是任务是否完成、过程是否健康、改动是否真的带来整体提升。

## 这页解决什么问题

这页用来建立一个稳定评估框架，避免把 Agent 质量误判成“试了几个例子感觉还行”。

## 选用原则

收录到这页的内容，优先满足以下条件：

- 能帮助区分结果评估、过程评估、对比实验和生产反馈
- 能把“回答器评测”与“任务执行系统评测”区分开
- 能支持持续迭代，而不是一次性打分

## 核心内容

### 为什么 Agent 评估和模型评测不同

一旦系统开始调用检索、工具、工作流或多代理机制，评估对象就从“回答质量”变成“任务执行系统质量”。

只看最终答案会掩盖三个问题：

- 结果正确，但过程低效或脆弱
- 结果错误，却不知道失败发生在哪一层
- 小改 prompt、模型或工具配置后，整体行为已变化但无从感知

### 四层评估框架

1. 结果评估：最终是否完成任务、是否相关、是否忠实、是否有幻觉
2. 过程评估：工具是否选对、顺序是否合理、是否存在冗余步骤
3. 对比实验：比较不同 prompt、模型、工具配置或 agent 结构
4. 生产反馈：线上监控、人工反馈、回退检测、长期质量漂移

### 结果评估回答什么问题

如果只看最终交付物，这个系统对用户是否可用？

适合：

- RAG 问答
- 摘要、分类、抽取
- 单轮或短链路 Agent

### 过程评估回答什么问题

这个 Agent 的做事方式是否合理？

重点看：

- 工具调用正确性
- 轨迹与参考流程是否偏差过大
- 是否在错误节点循环
- 是否存在不必要的步骤和成本

### 对比实验的价值

评估不应停留在“绝对好不好”，还要能比较：

- 不同模型
- 不同 system prompt
- 不同检索策略
- 单 Agent 与多 Agent

### 生产反馈的必要性

离线数据集无法完全覆盖真实流量。真正稳定的 Agent 系统，需要线上监控质量、成本、延迟和失败类型。
