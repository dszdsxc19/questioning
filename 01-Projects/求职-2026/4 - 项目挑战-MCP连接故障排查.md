# 最难忘的技术挑战

## 【背景与挑战】

在搭建 Agent 的底层基建、研发类似 MCP 的跨终端调度机制时，我们需要编写一个 MCP Server 来实现跨工具的调度交互。当时我们的后端服务是多实例部署的。在用 Cursor 和 Trae 等工具进行联调时，遇到了一系列非常棘手的问题：连接经常无故失败，同时观察到打向 AI 网关的请求并发数出现了异常的飙升。

## 【排查过程】

**第一阶段：陷入框架与代码的局部视角**

起初，我的排查思路局限在 MCP Server 的业务代码和实现框架里。我仔细比对了 MCP 官方关于 SSE 和 Streamable HTTP 协议的实现细则，确认出问题的链路是基于传统的 SSE 协议。前期我的排错方向走偏了，我使用了网络抓包工具去死磕 Cursor 发起的 HTTP 请求体，逐字逐句核对 JSON-RPC 的数据格式和参数传递，试图找出应用层格式校验失败的证据。经过反复比对，请求参数没有任何问题，局部代码排查彻底陷入僵局。

**第二阶段：转换思路，从抓包到日志分析**

既然应用层代码和请求体都没错，我决定跳出既有的前端和代码视角。我登录到集群去查看应用容器的运行状况，发现在单个 Pod 里的运行日志是不连续的、呈片段化的。这个关键线索瞬间触发了我对多实例部署形态的警觉。

**第三阶段：深挖协议底层的架构冲突**

结合残缺的日志和底层协议规范，我推导出了引发故障的致命交叉点。早期的 MCP 通信规范在使用 SSE 时，强制要求客户端访问两个完全独立的端点：一个是 GET 端点用于建立持久化的单向事件流，另一个是专门的 POST 端点用于发送 JSON-RPC 消息或心跳包。

由于 AI 网关主要是由团队里的资深同事负责维护，我前期忽略了网关层面的配置检查。后端的 Agent 处于多实例运行状态，且网关完全缺乏会话亲和配置。

## 【解决与落地】

我得出了明确的结论：客户端建立长连接的 GET 请求打到了实例 A，而紧随其后的后续消息 POST 请求，被负载均衡的轮询策略无情地分配到了实例 B。实例 B 的内存里根本没有这个客户端的会话上下文，直接判定请求非法，最终导致连接强制崩溃。客户端断连后不断重试，又进一步引发了网关请求量的飙升。

定位到根因后，我立刻联系负责网关的同事沟通验证。我们在网关层配置了基于 IP 或特定 Header 的会话粘滞策略，确保属于同一个大模型会话的连续网络请求始终被路由到同一个后端实例。调整网关策略后，流式请求立刻恢复稳定，彻底消除了连接中断现象，保障了后续业务团队在一天内顺利完成接入。

## 【总结与反思】

这次排障经历极大拓宽了我的技术视野。在现代前后端分离和云原生架构下，研发工程师排错的眼光必须超越单一的浏览器沙箱环境或业务代码逻辑。很多看似灵异的代码报错，其根源往往深藏在基础设施、网络协议栈以及网关路由的调度策略中。这也促使我在后续主导中大型项目的技术选型时，会更加注重端到端的架构一致性考量。
