## 概述

从外呼平台拿到实时通话数据流，VAD 检测对话，FunASR 转为对话文本流。Mastra 编排外呼业务状态机，对应节点和工作流。

---

## 一、LangGraph 状态机设计

### 1. 状态记忆 (State)

在 LangGraph 中，首先需要定义一个全局的状态对象。这个对象伴随整通电话，记录提取到的结构化信息：
- 用户当前意图
- 客户抗拒点
- 是否已报价
- 当前的通话轮次

每次对话回合结束，系统都会更新这个状态字典。

### 2. 业务节点设计 (Nodes)

将标准的外呼话术抽象为图的节点。每个节点内部都是一个独立的 LLM 调用，拥有专属且精简的设定。

| 节点 | 功能 |
|------|------|
| **开场白节点** | 只负责礼貌问候并确认对方身份 |
| **意图识别节点** | 分析客户刚才的话，提取核心诉求并更新到状态字典中 |
| **异议处理节点** | 专门配置用于克服客户顾虑的专业话术和产品知识库 |
| **促单节点** | 当全局状态显示客户疑虑已消除时，主动引导客户确认最终意向 |

### 3. 条件路由流转 (Conditional Edges)

这是图框架的灵魂所在。在一个节点执行完毕后，系统会触发路由函数。代码结合 LLM 的判断，根据当前的最新状态，决定下一步走向哪个节点。

**示例**：系统正处于"开场白节点"，客户回答"你们的产品太贵了"。路由节点分析出"价格抗拒"这一状态，随后精准地将对话权限移交给"异议处理节点"，由该节点生成关于性价比的针对性回复。

## 二、架构分层

### 1. 消息解耦层 (解决负载痛点)

> 状态：没做

- **工具**：Redis Pub/Sub、RabbitMQ 或 Apache Kafka
- **逻辑**：为了解决负载问题，AI 后端在生成识别文本和回复文本时，仅仅将这些数据打包成轻量级的 JSON，发布到 Redis 的指定频道中，随后立刻释放资源，绝不持有任何与前端的连接。

### 2. 监控展示层 (前端旁路大屏)

- **工具**：Node.js 网关 + Next.js / React 前端应用
- **逻辑**：部署一个专门处理高并发连接的轻量级 Node.js 服务。它负责订阅 Redis 中的数据，并维护与前端浏览器的 WebSocket 连接。前端页面使用 Next.js 等框架搭建交互 UI，接收到 WebSocket 推送的消息后，实时渲染出不同用户的对话气泡。

## 三、实时优化策略

### 1. LLM 响应优化

#### A. 先决策后措辞
把 LLM 从"逐字说话"改成"先出骨架再说"。

#### B. 允许"先发占位句"，再补全细节
降低首字延迟，提升用户体验。

#### C. 预生成和缓存
对外呼话术，很多节点输出高度模板化（开场白、身份确认、常见异议的第一句）。可以在进入节点时就预生成候选话术放在缓存里：
- 用户一句话结束后，只需要选一个最匹配的候选句
- 复杂情况再触发实时生成

> 不是一定要基于规则的。

### 2. 声学与语义双重校验 (Semantic Endpointing)

为了做到像真人一样反应敏捷，现代高级外呼系统会结合文本的语义完整度来做综合判断。

- **双重判断逻辑**：系统把 VAD 的静音阈值大幅调低（例如 400 毫秒）。当触发这个极短的静音时，立刻利用 FunASR 输出的标点符号预测结果，或者外挂一个极轻量级的本地文本分类模型，来评估当前这句话在语法结构上是否完整。
- **动态响应**：
  - 如果语义已经完整（比如"你们这个套餐多少钱。"），系统立刻截断并送给大模型
  - 如果语义明显不完整（比如"我觉得吧，那个……"），系统会继续保持聆听状态，静音计时器重置，等待用户补全后续内容

### 3. 过滤与拦截机制 (Backchanneling)

在真实的电话沟通中，客户经常会在 AI 持续输出话术时无意识地发出"嗯"、"哦"、"对的"等短促的捧哏词。

- **拦截策略**：收到这类极短且无明确意图的识别结果后，系统绝不应该将其送入 LangGraph 触发大模型重新推理。我们可以直接在接收 ASR 文本的关口加一层简单的正则或规则过滤。遇到毫无实际意图的语气词，直接丢弃，同时保持当前 AI 语音流的继续播放。
