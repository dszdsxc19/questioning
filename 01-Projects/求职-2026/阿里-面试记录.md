## 📌 题目一：Cookie 机制与属性

### 面试官问法
> "讲一下 Cookie，有哪些属性，分别是什么作用？"

### ✅ 参考回答

Cookie 是服务端通过 `Set-Cookie` 响应头写入浏览器的键值对，后续请求会自动携带在请求头 `Cookie` 中。

**核心属性：**

```
Set-Cookie: name=value; Domain=.example.com; Path=/; Expires=Wed, 09 Jun 2030 10:18:14 GMT; Max-Age=86400; Secure; HttpOnly; SameSite=Lax
```

| 属性 | 作用 |
|------|------|
| `Domain` | 指定哪些域名可携带此 Cookie（含子域） |
| `Path` | 限制 Cookie 适用的路径范围 |
| `Expires / Max-Age` | 持久化 Cookie 的过期时间；不设置则为 Session Cookie（关窗消失） |
| `Secure` | 只在 HTTPS 下传输 |
| `HttpOnly` | 禁止 JS 读取（`document.cookie`），防 XSS 窃取 |
| `SameSite` | 控制跨站请求是否携带 Cookie：`Strict`（完全禁止）/ `Lax`（导航可带）/ `None`（必须配合 `Secure`） |

**两类 Cookie：**
- **Session Cookie**：无 `Expires`/`Max-Age`，浏览器关闭即删除
- **持久化 Cookie**：有过期时间，存在磁盘

## 📌 题目二：SSE 项目痛点

### 面试官问法
> "你在项目里用了 SSE，遇到了什么痛点？是怎么解决的？"

### ✅ 参考回答（结构化）

**先说 SSE 是什么：**

SSE（Server-Sent Events）是基于 HTTP 的单向推送协议。服务端通过 `Content-Type: text/event-stream` 持续推送数据，客户端通过 `EventSource` API 接收。

**真实痛点（按轻重排）：**

#### ❶ 连接管理 & 断线重连
- 浏览器对同一域名有 **6 个 HTTP/1.1 并发连接**限制，SSE 会占用一个，多标签页会相互竞争
- 解决：升级到 **HTTP/2**（多路复用，不受此限制）；或用 **SharedWorker** 共享一个 SSE 连接

#### ❷ 服务端连接泄漏
- 客户端关闭页面后，服务端未必能及时感知（TCP 连接可能在 NAT 超时前一直保持）
- 解决：服务端定期发送心跳帧 `data: ping\n\n`，若写入失败则清理连接；配合 `req.on('close')` 回调

#### ❸ 消息重放（断线续传）
- SSE 自带 `Last-Event-ID` 头，断线重连时浏览器自动带上最后收到的事件 ID
- 服务端需要根据此 ID 从消息队列/数据库中重放未送达的消息

#### ❹ 鉴权问题
- `EventSource` 不支持自定义请求头，无法直接带 Bearer Token
- 解决方案：① URL 参数传 token（有安全风险，需短期 token）；② 依赖 Cookie 认证；③ 改用 fetch + ReadableStream 手动实现 SSE

#### ❺ 与 WebSocket 的取舍
| 对比维度 | SSE | WebSocket |
|----------|-----|-----------|
| 通信方向 | 单向（服务端→客户端） | 双向 |
| 协议 | HTTP | WS 协议（需握手升级） |
| 断线重连 | 浏览器自动重连 | 需手动实现 |
| 防火墙穿透 | 好（HTTP） | 可能被拦截 |
| 适用场景 | 消息推送、AI流式输出 | 聊天、实时游戏 |

## 📌 题目四：RAG 分割——图片与文本的区别

### 面试官问法
> "你说你做了 RAG，文档里有图片和文字，这两种内容在分割和向量化时有什么区别？"

### ✅ 参考回答

#### 文本分割策略

| 策略 | 描述 | 适用场景 |
|------|------|----------|
| 固定大小分割 | 按字符/Token 数切分 | 简单场景，速度快 |
| 语义分割 | 按句子/段落边界 + 语义相似度 | 长文档，保持连贯性 |
| 递归分割 | 先按段落→再按句子→再按词 | LangChain 默认策略 |
| 滑动窗口 | chunk 之间有 overlap | 防止答案被截断在边界 |

#### 图片的处理（与文本的核心区别）

```
文本文档
  └── 文字块 → tokenize → text embedding（如 OpenAI text-embedding-3）

混合文档（PDF/PPT/Word）
  ├── 文字块 → 同上
  ├── 图片块 → 两种路线：
  │     ├── 路线A：OCR → 提取文字 → text embedding
  │     │         （适合：截图、图表中的文字）
  │     └── 路线B：CLIP / LLaVA → image embedding
  │               （适合：照片、图示、流程图）
  └── 表格块 → OCR + 结构解析 → markdown/json → text embedding
```

**关键差异：**

1. **语义空间不同**：文本 embedding 和 image embedding 来自不同的向量空间，检索时需要使用**多模态模型**（如 CLIP）将文本 query 和图片 embedding 投影到同一空间
2. **分块粒度不同**：文本可细粒度分块，图片通常整张作为一个 chunk（切图会破坏语义）
3. **上下文关联**：图片往往依赖相邻的文字说明（图注/标题），需要将图片与周围文字一起打包进同一 chunk

> 📎 详见 [[真实面经回答/03-RAG 多模态文档解析策略]]

## 📌 题目五：Cookie 与跨域

### 面试官问法
> "跨域的时候，Cookie 怎么传？有什么限制？"

### ✅ 参考回答

跨域请求默认**不携带 Cookie**，需要前后端同时配置：

#### 前端配置

```javascript
// fetch
fetch('https://api.example.com/data', {
  credentials: 'include'   // ← 必须
})

// axios
axios.get('https://api.example.com/data', {
  withCredentials: true    // ← 必须
})

// EventSource（SSE）无法设置，只能依赖 Cookie 自动携带
const es = new EventSource('https://api.example.com/stream')
// 需要服务端 SameSite=None; Secure 配置
```

#### 服务端配置（Node/Nginx 等）

```http
Access-Control-Allow-Origin: https://your-app.com   ← 不能是 *
Access-Control-Allow-Credentials: true               ← 必须为 true
```

> ⚠️ **当 `credentials: 'include'` 时，`Access-Control-Allow-Origin` 不能为 `*`**，必须明确指定域名。

#### Cookie 的 SameSite 限制

| SameSite 值 | 跨站请求能带 Cookie 吗？ |
|-------------|--------------------------|
| `Strict` | ❌ 完全不带 |
| `Lax`（默认） | ⚠️ 仅顶层导航的 GET 可带（`<a>` 跳转） |
| `None` | ✅ 可以，但**必须同时设置 `Secure`**（即 HTTPS） |

**所以跨域传 Cookie 的完整要求：**
1. Cookie 设置了 `SameSite=None; Secure`
2. 前端请求设置 `credentials: 'include'`
3. 服务端响应头 `Access-Control-Allow-Origin` 为具体域名 + `Access-Control-Allow-Credentials: true`

## 📌 题目六：百万级流量，前端和后端如何设计？

### 面试官问法
> "假设你的产品突然有 100 万并发用户，前端和后端分别怎么设计？"

### ✅ 参考回答（分层展开）

#### 前端层

```
用户
  │
  ├── CDN（静态资源）
  │     ├── HTML / JS / CSS / 图片全部走 CDN
  │     ├── 内容寻址缓存（contenthash 文件名）
  │     └── 全球边缘节点，就近访问
  │
  ├── 首屏优化
  │     ├── 代码分割（动态 import）
  │     ├── 懒加载（路由懒加载、图片懒加载）
  │     └── SSR / SSG（减少白屏时间，降低服务器压力）
  │
  └── 缓存策略
        ├── 强缓存（Cache-Control: max-age）用于有 hash 的静态资源
        ├── 协商缓存（ETag / Last-Modified）用于入口 HTML
        └── Service Worker 离线缓存
```

#### 网关 / 接入层

```
CDN / DNS 负载均衡
  │
  └── API Gateway（Nginx / Kong / Traefik）
        ├── 限流（令牌桶/漏桶）：防止瞬时流量击穿
        ├── 熔断降级：下游服务故障时返回兜底数据
        ├── 鉴权：JWT / OAuth 统一处理
        └── 路由：将请求分发到微服务
```

#### 后端层

```
API Gateway
  │
  ├── 微服务（横向扩展）
  │     ├── 无状态设计：Session 外置到 Redis
  │     └── 水平扩缩容（K8s HPA）
  │
  ├── 缓存层
  │     ├── Redis（热点数据、用户 Session、计数器）
  │     └── 本地缓存（Caffeine）减少 Redis 压力
  │
  ├── 数据库层
  │     ├── 读写分离（主库写、从库读）
  │     ├── 分库分表（ShardingSphere / Vitess）
  │     └── 连接池（HikariCP）
  │
  └── 异步解耦
        ├── 消息队列（Kafka / RabbitMQ）
        │     └── 削峰填谷：流量高峰时排队处理
        └── 异步任务（Email、报表生成等）
```

#### 高可用保障

| 手段 | 目的 |
|------|------|
| 多机房/多可用区部署 | 防单点故障 |
| 熔断（Hystrix / Sentinel） | 下游故障隔离 |
| 限流（Rate Limiting） | 保护服务不被打垮 |
| 降级（Fallback） | 非核心功能关闭，保核心链路 |
| 监控告警（Prometheus + Grafana） | 快速发现和定位问题 |

## 🔍 复盘与反思

### 问题一：答题没有条理

**现象**：面试官问一个问题，脑子里有东西，但讲出来是"散的"，东一句西一句。

**根本原因**：缺少"先给结论，再展开"的答题习惯。

**解决方法 - 答题模板**：

```
1. 先给出一句话结论（定义/核心观点）
2. 再分 2-3 点展开（用"第一...第二...第三..."）
3. 最后举一个具体例子或说明适用场景
```

**示例对比：**

❌ 差的回答：
> "Cookie 就是...存在浏览器里的，然后它有很多属性，比如 Secure，还有 HttpOnly，HttpOnly 就是...JS 读不到，然后 SameSite 是用来控制跨站的..."

✅ 好的回答：
> "Cookie 是服务端下发、浏览器自动携带的状态存储机制。它有几个核心属性：第一，安全类属性 `HttpOnly` 防止 JS 读取，`Secure` 限制只在 HTTPS 传输；第二，作用域属性 `Domain` 和 `Path` 控制哪些请求会携带；第三，`SameSite` 控制跨站携带策略，分 Strict、Lax、None 三档。"

---

### 问题二：回答太啰嗦

**现象**：一句话能说清楚的事，说了一分钟。

**训练方法**：
- 每道题先写下关键词骨架，再口述
- 计时练习：限 90 秒内说完一道中等题
- 录音回听，找"废话句型"（"就是说"、"然后呢"、"我觉得"）
