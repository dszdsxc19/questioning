# Redis 六种数据类型与常见用法

## 一、核心数据类型与常见用法

### String（字符串）
- **定义**：最基础的键值对结构
- **底层实现**：SDS（Simple Dynamic String）
- **典型场景**：
  - 常规计数（点赞数、阅读量）
  - 验证码存储
  - 分布式系统中的全局序列号生成
  - 单值数据缓存
- **常用命令**：`SET`、`GET`、`INCR`、`DECR`、`SETEX`

### List（列表）
- **定义**：双向链表结构
- **底层实现**：quicklist（3.2+）
- **典型场景**：
  - 轻量级的消息队列
  - 最新动态的时间线拉取
- **常用命令**：`LPUSH`、`RPUSH`、`LPOP`、`RPOP`、`LRANGE`

### Hash（哈希）
- **定义**：键值对集合
- **底层实现**：dict + ziplist/intset
- **典型场景**：
  - 存储对象的详细属性（用户资料、商品详情）
  - 支持部分字段的局部读取与修改
  - 有效节省网络开销
- **常用命令**：`HSET`、`HGET`、`HMGET`、`HGETALL`、`HINCRBY`

### Set（集合）
- **定义**：无序且唯一的集合
- **底层实现**：dict + intset
- **典型场景**：
  - 共同好友推荐
  - 用户兴趣标签筛选
  - 交集、并集、差集计算
- **常用命令**：`SADD`、`SREM`、`SINTER`、`SUNION`、`SDIFF`

### ZSet（有序集合）
- **定义**：带有排序权重的集合
- **底层实现**：skiplist + dict + ziplist
- **典型场景**：
  - 实时积分排行榜
  - 延时任务队列
  - 范围查询（按分数/排名）
- **常用命令**：`ZADD`、`ZRANGE`、`ZREVRANGE`、`ZRANK`、`ZSCORE`

### 进阶类型
- **Bitmaps**：海量用户的活跃度或签到统计
- **HyperLogLog**：极低内存消耗的百万级 UV 估算
- **GEO**：地理坐标查询（附近的人/地点）
- **Streams**：持久化消息队列功能

## 二、Redis 六大常见用法

### 1. 缓存层（Caching）
**原理**：将高频访问且变动极少的数据放在内存中

**优势**：
- 极大减轻后端关系型数据库的压力
- 显著提升系统整体响应速度

**典型场景**：
- 热点新闻文章
- 商品详情
- 临时验证码
- 频繁查询的系统配置信息

### 2. 限流防御（Rate Limiting）
**原理**：利用原子递增操作结合过期时间

**实现方式**：
```
# 伪代码
count = INCR key
IF count == 1:
    EXPIRE key 60
IF count > threshold:
    REJECT
```

**典型场景**：
- 短信发送频率限制
- API 接口调用频率控制
- 防止密码被暴力破解

### 3. 排行榜与计数器（Leaderboards and Counters）
**原理**：利用 ZSet 的排序能力和原子自增指令

**优势**：
- 高并发下的安全计数
- 实时排序功能

**典型场景**：
- 游戏积分排行榜
- 微博热搜榜
- 文章/视频的实时阅读量统计

### 4. 分布式会话管理（Session Management）
**原理**：将 Session 数据统一集中存储在 Redis 中

**优势**：
- 实现无状态的 Web 服务部署
- 跨服务器共享用户登录状态

**典型场景**：
- 大型电商网站的用户登录状态保持
- 单点登录（SSO）系统

### 5. 消息队列（Message Queues）
**原理**：基于 List 或 Streams

**注意**：重量级场景推荐 Kafka/RabbitMQ

**典型场景**：
- 异步发送邮件通知
- 后台耗时任务分发
- 简单的在线聊天室

### 6. 分布式锁（Distributed Locks）
**原理**：通过 `SETNX` 或 Redlock 算法

**核心要点**：
- 确保同一时刻只有一个服务节点执行
- 需要处理锁过期、死锁等问题

**典型场景**：
- 电商秒杀活动的超卖防护
- 定时任务的单节点防重复执行

## 三、面试高频常考点

### 1. 缓存异常问题及解决方案

#### 缓存穿透
**问题**：恶意查询大量根本不存在的数据，请求直接打穿缓存直奔数据库

**解决方案**：
- 引入布隆过滤器
- 缓存空值并设置较短过期时间
- 接口层校验

#### 缓存击穿
**问题**：某个极高并发的热点 Key 突然过期，大量请求瞬间全部打到数据库

**解决方案**：
- 加互斥锁重建缓存
- 热点数据设置为逻辑永不过期
- 异步刷新缓存

#### 缓存雪崩
**问题**：大量 Key 在同一时间集体过期，或 Redis 节点大规模宕机

**解决方案**：
- 给过期时间加上随机抖动值
- 部署高可用集群（哨兵/集群模式）
- 多级缓存架构

### 2. 持久化机制

| 方案 | 原理 | 触发时机 | 优点 | 缺点 |
|------|------|----------|------|------|
| **RDB** | 内存快照 | 定时/手动 | 文件紧凑、恢复快 | 可能丢失数据、fork 阻塞 |
| **AOF** | 追加写命令 | 每秒/每次写 | 数据完整性好 | 文件大、恢复慢 |
| **混合持久化** | RDB + AOF | 重写时 | 兼顾速度和完整性 | 4.0+ 版本 |

### 3. 内存淘汰策略

**算法对比**：
- **LRU**（Least Recently Used）：最近最少使用
- **LFU**（Least Frequently Used）：最不经常使用

**Redis 策略**：
- `noeviction`：不淘汰，写入报错
- `allkeys-lru`：对所有 key 使用 LRU
- `allkeys-lfu`：对所有 key 使用 LFU（4.0+）
- `volatile-lru`：对设置了过期时间的 key 使用 LRU
- `allkeys-random`：随机淘汰
- `volatile-ttl`：淘汰即将过期的 key

**注意**：Redis 使用近似算法实现 LRU/LFU，非精确实现

### 4. 高可用与集群架构

#### 主从复制
- **全量同步**：slave 发送 SYNC，master 发送 RDB
- **增量同步**：复制积压缓冲区（replication backlog）
- **问题**：master 宕机需要手动切换

#### 哨兵（Sentinel）
- **功能**：监控、自动故障转移、配置中心
- **流程**：主观下线 → 客观下线 → 选举领导者 → 故障转移
- **Quorum**：多数哨兵同意才判定下线

#### Redis Cluster
- **分片机制**：哈希槽（16384 个槽）
- **路由逻辑**：CRC16(key) % 16384
- **高可用**：每个主节点有从节点
- **MOVED/ASK**：重定向机制

## 四、WWTS 面试回答模板

当被问及 Redis 数据结构时，可以按此框架回答：

1. **What**：明确指出是哪种数据类型，说明其基本特性
2. **Why**：解释为什么用这个数据结构，有什么优势
3. **Tradeoffs**：分析其限制、内存消耗、适用边界
4. **Scenarios**：举出具体项目中的使用案例
