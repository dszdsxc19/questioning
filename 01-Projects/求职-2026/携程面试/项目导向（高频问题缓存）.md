## Q1: 介绍一下 RAG 的流程，以及你做过哪些优化？

参考: [# Advanced RAG Techniques: Elevating Your Retrieval-Augmented Generation Systems](https://github.com/NirDiamant/RAG_Techniques)

### 口述版（1-2 分钟）

#### 1. 一句话（本质）
RAG 是**检索增强生成**，让 LLM 在回答时先检索外部知识库，再生成答案，解决模型知识过时和幻觉问题。

#### 2. 核心机制（怎么工作的）
三个阶段：
- **索引阶段**：文档分块 → 向量化 → 存入向量数据库
- **检索阶段**：问题向量化 → 向量相似度检索 → 找到最相关的 k 个 chunk
- **生成阶段**：检索到的上下文 + 问题 → 组装 prompt → LLM 生成答案

#### 3. 怎么实现（工程层）
我主要做了三个优化：

**第一，混合检索**。纯向量检索对专业术语效果不好，我加了 BM25 关键词检索，和向量检索融合召回，准确率提升了约 15%。

**第二，重排序**。先快速召回 50 个候选，再用交叉编码器 rerank 到 top 5，虽然增加了 100ms 延迟，但相关性明显提升。

**第三，分块优化**。尝试了固定大小、语义分块、父子文档几种策略，最后发现父子文档效果最好——小 chunk 用于精准检索，大 chunk 提供完整上下文。

#### 4. 项目经验（加分项）
在我们项目中，用户问的都是垂直领域的专业问题，纯 RAG 容易检索不到相关内容。我加了查询重写，让 LLM 先把用户问题扩展成多个查询变体，再并行检索，最后合并去重，这样召回率提升了 20% 左右。

### 技术细节备忘（供追问参考）

**其他优化手段**（按优先级）：
- 查询重写 / HyDE
- 上下文压缩
- Self-RAG（自我反思）
- 微调 embedding 模型
- GraphRAG（知识图谱）

**常见追问应对**：
- Q: 为什么选这个 embedding 模型？
  - A: 考虑了中文支持、向量维度、检索速度，在几个开源模型上做了 ablation 测试

- Q: 怎么评估 RAG 效果？
  - A: 用 RAGAS 框架，看 faithfulness（忠实度）和 answer relevance（答案相关性）

- Q: 检索失败怎么办？
  - A: 加了 fallback 机制，如果检索相关性低于阈值，会直接让模型回答或提示用户问题超出范围

## Q2: 怎么对 PDF 表格等结构化场景做 RAG？

### 口述版（1-2 分钟）

#### 1. 一句话（本质）
表格 RAG 的核心是**结构化信息的保留与提取**，不能简单 OCR，需要识别表格的行列关系，让 LLM 能理解结构化数据。

#### 2. 核心机制（怎么工作的）
三种主流方案：

**方案一：LLM 视觉理解**
- 直接把表格当图片输入多模态 LLM（如 GPT-4V）
- 优点：能理解复杂表格
- 缺点：成本高、速度慢

**方案二：OCR + 结构识别**
- 图片 → OCR 提取文本 → 表格结构识别模型（如 Table-Transformer）
- 识别出单元格位置 → 重构表格
- 优点：准确率高，可本地化

**方案三：工具化方案（工程常用）**
- 边框表格：用线检测（Camelot lattice 模式）
- 无边框表格：用文本对齐分析（Camelot stream 模式）
- 输出 Markdown/HTML 格式给 LLM

#### 3. 怎么实现（工程层）
我采用了**分场景策略**：

**第一，文档预处理**。先判断 PDF 类型：
- 数字原生 PDF（有文本层）：直接提取，用 pdfplumber 解析表格结构
- 扫描 PDF（纯图片）：走 OCR 流程

**第二，表格检测与识别**。
对于有边框表格，用 Camelot 的 lattice 模式，基于线条检测识别表格边界，准确率很高；
对于无边框表格，用 stream 模式，通过文本对齐和间距推断表格结构。

**第三，结果标准化**。
无论用哪种方式，最后都统一输出 Markdown 表格格式，这样 chunk 的时候能把相关行放在一块，检索时能保持上下文完整性。

#### 4. 项目经验（加分项）
我们项目中有很多财务报表和技术规格书的 PDF，一开始直接 text 提取，表格里的数字全乱了，检索"某个产品参数"时找不到对应数据。后来用 Camelot 做了表格解析，把每一行转换成"表头+值"的自然语言描述，比如把"CPU: Apple M1"转换成文本，这样检索和生成都准确多了。

## Q3: LEFT JOIN 中条件写在 ON 和 WHERE 有什么区别？

### 口述版（1-2 分钟）

#### 1. 一句话（本质）
**ON 条件在 JOIN 时过滤，WHERE 条件在 JOIN 后过滤**——写在 ON 里能减少中间结果集，性能更好。

#### 2. 核心机制（怎么工作的）

**写在 ON 里：**
```sql
SELECT * FROM A
LEFT JOIN B ON A.id = B.a_id AND B.status = 1;
```
执行流程：
- 先做 LEFT JOIN
- 在 JOIN 过程中对 B 进行过滤
- **A 表的数据一定全部保留**

**写在 WHERE 里：**
```sql
SELECT * FROM A
LEFT JOIN B ON A.id = B.a_id
WHERE B.status = 1;
```
执行流程：
- 先 LEFT JOIN（生成完整中间结果）
- 再整体过滤结果
- **会过滤掉 A 表中不匹配的行**（相当于变成了 INNER JOIN）

**SQL 执行顺序**：FROM → JOIN → ON → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT

#### 3. 怎么优化（工程层）

**原则**：能放在 ON 的过滤条件不要放到 WHERE

| 场景 | ON 条件 | WHERE 条件 |
|------|---------|------------|
| LEFT JOIN | 过滤右表，保留左表 | 整体过滤，可能丢失左表数据 |
| INNER JOIN | 效果相同，推荐 ON | 效果相同，推荐 ON |
| 性能 | 减少中间结果集 | 先全量 JOIN 再过滤 |

**关键差异示例**：
```sql
-- 场景：查询所有用户及其有效订单
-- ✅ 写在 ON：用户会全部保留
SELECT u.*, o.* FROM users u
LEFT JOIN orders o ON u.id = o.user_id AND o.status = 1;

-- ❌ 写在 WHERE：没有有效订单的用户会丢失
SELECT u.*, o.* FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE o.status = 1;
```

#### 4. 项目经验（加分项）

在一个订单查询接口中，原本写在 WHERE 的条件导致没有符合条件的用户被过滤掉，业务逻辑出错。改成 ON 条件后，不仅修复了 bug，由于减少了中间结果集，查询时间从 300ms 降到 120ms。

### 技术细节备忘（供追问参考）

**一、慢查询优化**

#### 本质
慢查询优化 = 减少扫描数据量 + 提高索引命中率 + 优化执行计划

#### 核心机制

慢的原因通常是：
- 没走索引 → 全表扫描
- 回表过多
- JOIN 数据量过大
- 排序/分组没索引

**工具：EXPLAIN SELECT ...**
关注：
- `type`：ALL（全表） / index（全索引） / ref（索引查找） / range（范围）
- `rows`：扫描行数
- `extra`：Using filesort / Using temporary

#### 优化手段

**（1）加索引**
```sql
-- WHERE 条件字段
WHERE user_id = ?
-- 👉 加 (user_id) 索引

-- 联合索引注意最左前缀
WHERE user_id = ? AND status = ?
-- 👉 加 (user_id, status) 索引
```

**（2）避免索引失效**
```sql
-- ❌ 字段运算
WHERE age + 1 = 20

-- ✅ 常量运算
WHERE age = 19

-- ❌ 函数包裹
WHERE YEAR(create_time) = 2024

-- ✅ 范围查询
WHERE create_time >= '2024-01-01' AND create_time < '2025-01-01'
```

**（3）优化分页**
```sql
-- ❌ 深分页：越往后越慢
LIMIT 100000, 10

-- ✅ 游标分页：基于主键
WHERE id > last_id LIMIT 10
```

**（4）覆盖索引（减少回表）**
```sql
-- 创建联合索引
ALTER TABLE orders ADD INDEX (user_id, status, create_time);

-- 查询只走索引，不回表
SELECT user_id, status FROM orders WHERE user_id = ?;
```

---

**二、JOIN 优化**

#### 本质
JOIN 优化的核心是减少参与 JOIN 的数据量，并保证连接字段有索引。

#### 核心机制

MySQL JOIN 本质是 **Nested Loop Join（嵌套循环）**：
- 外表一行一行扫描
- 内表通过索引查匹配

#### 优化手段

**（1）小表驱动大表**
```sql
-- ✅ 小表在外
SELECT * FROM small s
JOIN big b ON s.id = b.sid;
```

**（2）JOIN 字段加索引**
```sql
ON A.id = B.a_id
-- 👉 B.a_id 必须有索引
```

**（3）提前过滤（关键）**
```sql
LEFT JOIN B
  ON A.id = B.a_id AND B.status = 1
-- 👉 减少 JOIN 数据量
```

**（4）控制 JOIN 数量**
- 超过 3 个 JOIN 考虑拆分查询
- 或用冗余字段减少 JOIN

---

**常见追问应对**：

- Q: 什么时候用覆盖索引？
  - A: 查询字段都在索引里，不需要回表，性能最好

- Q: 怎么判断走没走索引？
  - A: EXPLAIN 的 type 列，ref/range 最好，ALL 是全表扫描

- Q: 联合索引的顺序怎么设计？
  - A: 区分度高的放前面，结合查询条件频率设计

## Q4: 大模型和 agent 的评估你们是怎么做的？

### 口述版（1-2 分钟）

#### 1. 一句话（本质）
大模型和 agent 评估的本质，不是看模型答得漂不漂亮，而是看它有没有稳定地对业务产生价值；其中大模型更偏答案质量，agent 还要额外评估过程是否正确、系统是否可落地。

#### 2. 核心机制（怎么工作的）
我们实际把评估拆成三层，这样特殊情况最少，也最贴近线上：

- **Outcome（结果层）**：任务有没有完成，回答对不对，是否真的推进了业务目标。这一层是核心。
- **Process（过程层）**：主要是 agent 特有的，比如工具调得对不对、规划是否合理、多轮轨迹有没有跑偏、有没有出现循环或无效步骤。
- **System（系统层）**：看 latency、token、cost、失败率、工具调用次数这些工程指标。企业最后能不能上线，往往卡在这一层。

如果只评最终答案，会漏掉很多隐性错误。很多 agent 表面答对了，但工具调用是错的、路径是脏的，这种东西线上迟早出事。

#### 3. 怎么实现（工程层）
我们做法比较直接，就是先建业务数据集，再做自动化评估 pipeline，最后用线上数据闭环：

- **先做离线数据集**：不是拿通用 benchmark，而是从真实对话日志里抽样，覆盖成功案例、失败案例和边界 case，保证评估贴近真实业务。
- **再定指标体系**：大模型侧主要看 correctness、relevance、hallucination；agent 侧再补 tool correctness、trajectory quality、task success、latency、cost。
- **自动评估分两层**：第一层是规则评估，比如是否命中正确知识、是否调用了对的工具；第二层是 `LLM-as-a-Judge`，去评估回答质量、是否符合策略、有没有幻觉。
- **最后做在线闭环**：通过 A/B test、人工抽检和真实转化数据，看不同 prompt、不同策略到底有没有带来业务收益。

本质上，大模型 eval 更像“答案质量评估”，agent eval 则是“结果 + 过程 + 系统”一起评，不然后面一定会被工程现实打脸。

#### 补充：评估里的取舍怎么做
这块面试里我一般会直接说，评估不是把所有指标都拉满，因为很多指标天然冲突，比如准确率更高往往意味着链路更长、模型更大、工具更多，最后延时和成本都会上去。

所以我们的做法不是追单一最优，而是分成两步：

- **先定优先级**：业务结果优先，也就是 task success 或转化相关指标必须先达标；如果结果不行，系统再快也没意义。
- **再设硬约束**：比如实时场景里，TTFT、总响应时间、失败率、成本都要有明确上限，超过阈值的方案直接淘汰。

实际落地时，常见取舍有三个：

- **效果 vs 延时**：复杂 agent、RAG、多工具调用通常能提升成功率，但会拉高延时。我们的做法是分层路由，简单问题直出，复杂问题再进 agent。
- **效果 vs 成本**：更强模型 judge 更准，但离线评估和线上推理成本都会上升。一般是核心链路用强模型，非核心链路做采样评估或降级。
- **准确率 vs 稳定性**：有些策略离线分数高，但线上波动大、失败率高。这个时候不能只看均值，要把 failure rate 和 fallback 情况一起看。

所以最后真正上线的，不是离线分最高的方案，而是**在业务收益、延时、成本、稳定性之间整体最优的方案**。这也是 agent eval 和传统模型评估最大的区别，它必须面对工程现实。

#### 4. 项目经验（加分项）
在我做 AI 销售外呼 Agent 的时候，我们后面专门补了 trajectory 和 tool correctness 评估。因为一开始只看最终结果，发现有些回答看起来没问题，但中间其实调错了工具，或者对话路径已经偏离销售目标。后来把过程指标和 TTFT、失败率一起纳入评估后，问题定位和策略迭代效率明显高很多，整个系统才真正能在线上稳定跑起来。

## Q5: 你在项目里是怎么使用 Redis 的？

### 口述版（1-2 分钟）

#### 1. 一句话（本质）
我在项目里用 Redis，本质上不是把它当数据库，而是把它当**高频读写路径上的低延迟基础设施**，核心目标就一个：**降低响应延迟，扛住高并发，同时接受弱一致性**。

#### 2. 主要用在哪些场景
我主要用了五类场景：

- **RAG 缓存**：对查询重写结果、召回结果、重排结果，或者最终答案做短 TTL 缓存，避免重复走 embedding、检索和 rerank。
- **高并发限流和削峰**：对用户请求、会话请求、接口级别流量做限流，避免大模型和下游工具被瞬时流量打爆。
- **动态 prompt 模板缓存**：把 prompt 配置、版本、开关、策略参数放 Redis，做到快速热更新，不用每次都查库。
- **工具调用结果缓存**：一些相对稳定但查询代价高的工具结果，比如价格、库存、规则配置、外部接口聚合结果，会缓存一层。
- **用户画像缓存**：把用户最近行为、偏好标签、上下文摘要放 Redis，减少每轮对话都查主存储。

这几个场景有一个共性：**都不要求强一致性，但都非常在意 RT**。所以 Redis 很适合。

#### 3. 具体怎么做（工程层）
我一般会先分清三类 key：

- **热点结果缓存**：比如 `rag:query_hash`、`tool:args_hash`
- **会话态数据**：比如 `session:user_id`、`profile:user_id`
- **控制类数据**：比如 `rate_limit:*`、`lock:*`、`prompt:template:*`

在实现上有几个重点：

- **缓存必须带 TTL**，避免脏数据无限存活。
- **热点 key 做随机过期时间**，比如 `base_ttl + random_jitter`，防止同一时间大面积失效。
- **限流用 Redis + Lua**，因为 incr + expire 分开写会有竞态，Lua 可以保证原子性。
- **查不到缓存时，不是所有请求都去打 DB / RAG**，而是用 `SET lock:key value NX PX 3000` 先抢锁，只放一个请求去回源，其它请求等待、快速失败，或者返回旧值。

#### 4. 缓存异常问题怎么处理
这个面试官很容易追问，我一般会直接分三种说：

- **缓存穿透**：查一个根本不存在的数据，请求每次都打到底层。我的做法是两层：
  - 对非法参数先做前置校验；
  - 对明确不存在的结果做**空值缓存**，给一个较短 TTL。

- **缓存击穿**：一个热点 key 过期瞬间，大量请求同时回源。处理方式就是你说的这套：
  1. 发现 key 不存在；
  2. 尝试 `setnx` 或 `SET lock:key value NX PX 3000` 加锁；
  3. 只有一个请求去查 DB / RAG；
  4. 其他请求等待、重试，或者直接返回旧值。

- **缓存雪崩**：大量 key 同时过期，底层被打崩。我的做法是：
  - TTL 加随机值，避免同一批 key 同时失效；
  - 多级降级，比如优先返回旧缓存、默认答案、兜底提示；
  - 对核心链路再叠加限流和熔断。

#### 5. 一致性怎么权衡
这个场景我会明确说，我们接受的是**弱一致性**，因为这些缓存大部分服务的是“更快响应”，不是“交易正确性”。

比如用户画像、prompt 模板、工具结果、RAG 结果，只要不是强事务场景，短时间不一致是可以接受的。因为如果为了强一致性把每次请求都打到 DB、向量库或外部系统，延迟和吞吐都会明显变差，这在对话系统里不划算。

所以我们的原则是：

- **能缓存就缓存**
- **能短暂旧一点就不要强求绝对新**
- **把强一致性留给真正不能错的链路**

### 技术细节备忘（供追问参考）

**一、为什么限流要用 Redis + Lua？**

因为分布式场景下，限流计数必须是原子的。比如固定窗口或滑动窗口，如果只是客户端先 `INCR` 再 `EXPIRE`，中间挂了就会出现脏状态。Lua 脚本能把读、改、写合成一次原子执行，简单直接，不容易出竞态。

**二、为什么锁用 Redis 原生命令，而不是自己拼？**

我会优先用：

```text
SET lock:key value NX PX 3000
```

原因是：
- `NX` 保证只有 key 不存在时才能加锁
- `PX` 直接带过期时间，避免拿到锁的线程挂死后形成死锁

释放锁时要校验 value，避免把别人的锁删掉。

**三、分布式锁在这里解决什么问题？**

不是为了做复杂分布式协调，而是为了**防止热点 miss 时的大量并发回源**。这是个很实用的点，别说玄学。

**四、RAG 缓存一般缓存什么？**

看业务链路，通常可以缓存：

- query rewrite 结果
- embedding 结果
- retrieval topK 结果
- rerank 结果
- 最终 answer

不是所有层都要缓存，关键看命中率和序列化成本。命中率很低的缓存就是垃圾缓存，只会增加复杂度。

**五、动态 prompt 模板为什么适合放 Redis？**

因为这类配置读取频繁、更新不算特别高，而且要求能快速生效。放 Redis 后，应用实例不需要重启，改模板能秒级生效，适合做 prompt 实验、灰度和版本切换。

**六、有哪些风险点？**

- 缓存 key 设计不稳定，命中率会很差
- value 太大，网络和序列化成本会吞掉收益
- 过度缓存，导致失效策略和一致性变复杂
- 锁粒度太粗，反而把并发压死
- Redis 成为单点热点，需要做热点隔离或降级

**七、面试里怎么收口？**

可以直接用这句话收：

> 我们这个项目里用 Redis，核心不是为了堆技术点，而是把高频链路上那些“贵但不要求强一致”的操作前移到内存层，目标一直很明确，就是降低响应延迟、控制并发冲击、让系统在线上更稳。

## Q6: 你在项目里是怎么使用消息队列 MQ 的？

### 口述版（1-2 分钟）

#### 1. 一句话（本质）
我在项目里用 MQ，本质上是为了把**非实时、重计算、易堆积**的工作从主链路拆出去，做到**异步化、削峰填谷和系统解耦**。不然所有事情都同步做，延迟会很难看，系统也扛不住高并发。

如果用更标准的面试话术概括，就是 MQ 解决的是三个经典问题：**解耦、异步、削峰**。

#### 2. 主要用了哪些场景
我一般重点说三个场景，够了，再多就散了。

**第一，通话后处理**，这是最核心的场景。

一通电话结束之后，后面其实还有很多重活：

- 通话转写整理
- LLM 复盘分析
- 质检打分
- 结构化信息抽取
- CRM 入库

这些如果全部同步做，会直接拖慢主流程，用户要等，系统峰值也会被打爆。

所以我们的改造思路很直接：

- 主流程只负责把“通话结束”这件事落库
- 然后发一条 MQ 消息
- 后面由不同消费者异步去做转写、分析、抽取、入库

这样主流程和后处理链路就解耦了。即使后面某个分析服务变慢，也不会卡死通话主链路。

**第二，日志和观测异步化。**

这个和你简历里的 tracing 能连起来说。比如：

- LLM 请求日志
- Tool 调用日志
- 用户行为日志
- tracing/span 相关事件

这些日志如果同步写数据库或 ES，主请求会被 IO 拖慢。所以我们会先异步写 MQ，再由日志消费链路统一落库、聚合、检索。

这样做的好处是：

- 降低主链路 RT
- 避免高峰期日志把业务库打挂
- 更方便后续做审计、问题排查和链路追踪

**第三，异步通知或非核心任务。**

比如外呼结果回写、标签更新、报表统计、消息通知，这些都不应该卡在用户请求里。原则很简单：**只要不是必须实时返回给用户的，都优先考虑异步化。**

#### 3. 价值怎么概括
我一般会把 MQ 的价值概括成三点：

- **系统解耦**：主流程只做核心动作，后处理独立扩展，新加消费者不用改主流程
- **异步提速**：原来串行调用的一堆非核心逻辑，改成发消息后异步执行，主接口 RT 会明显下降
- **削峰填谷**：高峰流量先进队列，消费者按系统承载能力平稳处理，避免瞬时洪峰直接打爆 DB 或下游服务

#### 4. 项目里我最看重什么
不是“有没有上 MQ”，而是三件事：

- **消息尽量不丢**
- **重复消息能处理**
- **有顺序要求的场景能保证顺序**

不把这三个说清楚，MQ 这块回答就是空的。

### 技术细节备忘（供追问参考）

**一、先把几个基础概念说清楚**

如果面试官先问基础，我会用最短的话解释：

- **Producer**：生产者，负责发消息的服务
- **Consumer**：消费者，负责处理消息的服务
- **Topic**：消息主题，可以理解成一类消息的逻辑分类，比如“通话后处理”“日志事件”
- **Partition**：Topic 下面的分区，用来提升吞吐和并发能力；消息通常不是直接进 Topic，而是进某个 Partition
- **Consumer Group**：消费者组，同一组内一个 Partition 通常只会被一个 Consumer 消费，用来实现横向扩展
- **Offset**：消费位点，表示某个 Consumer 消费到哪里了，方便继续消费和失败恢复

面试里讲 Topic 和 Partition 时，核心要说明两点：

- **Topic 解决的是消息分类问题**
- **Partition 解决的是吞吐、并发和局部顺序问题**

比如一个“通话后处理”Topic，里面可以按 `call_id` 或 `user_id` 把消息打到不同 Partition，这样既能并行处理，又能保证同一通电话的消息顺序。

**二、怎么保证消息尽量不丢？**

先说结论：工程上通常不是说“绝对 100% 不丢”，而是说**通过多层机制把丢消息风险压到足够低，并且有补偿手段**。真敢说绝对不丢，多半是在吹牛。

我一般从生产端、Broker、消费端三层说：

- **生产端**：
  - 业务数据先落库，再发消息，避免业务成功但消息没发出去没人知道
  - 更稳一点会用 **本地消息表 / Outbox 模式**：业务事务和待发送消息一起提交，再由后台任务投递 MQ
  - 生产者开启发送确认，确保 broker 确实收到了；如果返回异常就重试或转补偿流程

- **Broker 层**：
  - 消息持久化
  - 多副本 / 高可用部署
  - broker 落盘确认后再 ack 给生产者

- **消费端**：
  - 一定是**处理成功后再 ack / commit**
  - 消费失败不要直接吞，走重试、死信队列、人工补偿
  - 关键消息保留消费状态，方便对账和补偿

如果面试官追问“怎么做到接近 100% 不丢”，我会答：**Outbox + 发送确认 + Broker 持久化 + 消费端幂等 + 重试/死信 + 补偿对账**。这是比较完整的一套。

**三、消息重复怎么办？**

重复消息不是异常，是 MQ 系统的常态。因为只要你做重试，就必须接受“至少一次投递”带来的重复消费。

所以关键不是“避免重复”，而是**让消费逻辑幂等**。常见做法有：

- 给每条业务消息一个唯一业务 ID
- 消费前先查去重表 / Redis 去重 key
- 数据库更新用唯一索引、防重表、状态机约束
- 已处理过的消息直接跳过

比如 CRM 入库、通话分析结果回写，这类操作都必须做幂等，不然一重试就会重复写数据。

**四、顺序性怎么保证？**

先别上来就说“全局有序”，那是给自己找麻烦。真正该问的是：**到底是谁需要顺序？**

通常我们只保证**局部有序**，比如：

- 同一个用户
- 同一个会话
- 同一通电话

实现方式一般是：

- 按 `user_id`、`session_id`、`call_id` 做分区键
- 同一个 key 的消息路由到同一个 partition / queue
- 一个 partition 内由单消费者顺序消费

这样成本可控，也够解决业务问题。全局顺序通常吞吐太差，不值得。

**五、如果消息积压了怎么办？**

先判断是不是消费者有 bug 或下游变慢了。别一上来就加机器，那是糊弄。

如果是消费逻辑有问题，先修 bug，恢复消费能力；如果不是 bug，而是单纯消费速度跟不上，可以这样处理：

- 优化消费逻辑，比如单条处理改批量处理
- 扩分区 / 扩队列数
- 扩消费者实例，提高并行消费能力
- 如果已经积压很多，临时扩容做快速消费，消费完成后再回收资源

本质上，消息积压就是**生产速度大于消费速度**，解法也只有两种：要么让消费者更快，要么让流量进来得更平稳。

**六、通话后处理为什么特别适合 MQ？**

因为它天然就是“主流程结束后的一串重任务”：

- 耗时长
- 可并行

## Q7: 你们是怎么做延迟优化和并发调度的？

### 口述版（2-3 分钟）

#### 1. 一句话（本质）
这件事的本质，不是把所有任务一起跑，而是**围绕 TTFT 的关键路径做延迟预算**，把串行链路拆成并行 DAG，通过**异步化、优先级调度、缓存和降级**，优先保证首包稳定。

如果再压缩成一句更像面试回答的话，就是：

> 我们优化的不是平均响应时间，而是 P95 的首包延迟，所以所有调度策略都是围绕主链路做资源倾斜，非核心任务能并行就并行，来不及就降级或异步。

#### 2. 先把整体链路说清楚
我们的实时链路大致是：

`用户说话 -> WebRTC 传输 -> 流式 ASR -> Agent 编排 -> 并行跑 RAG / 用户特征 / 情绪识别 -> LLM 生成 -> streaming 返回给前端`

这里我会把任务拆成三类：

- **P0：主链路任务**
  - WebRTC 音频到达
  - ASR 增量识别
  - 意图识别
  - prompt 骨架组装
  - LLM 首 token 返回

- **P1：增强任务**
  - RAG 检索
  - 用户画像 / 用户特征加载
  - 情绪识别
  - 实时话术推荐

- **P2：非实时任务**
  - 埋点
  - 通话复盘
  - 质检分析
  - 日志异步落库

核心原则很简单：

- **P0 必须服务 TTFT**
- **P1 可以并发，但不能拖死主链路**
- **P2 必须彻底出主链路**

#### 3. TTFT 0.5s 是怎么拆出来的
我一般会直接给一个 latency breakdown，这样面试官会觉得你是真的量过。

一个比较稳定的拆法是：

- **WebRTC 传输**：50~80ms
- **ASR 增量识别**：120~150ms
- **意图识别 + prompt 骨架组装**：30~50ms
- **并行阶段**
  - RAG：80~120ms
  - 用户特征加载：20~40ms
  - 情绪识别：30~50ms
- **LLM TTFT**：180~250ms

所以主链路看的是：

`WebRTC + ASR + max(RAG, 用户特征, 情绪识别) + LLM TTFT`

如果按一个常见数字去讲，可以说：

- ASR：150ms
- RAG：100ms
- LLM TTFT：200ms

再加上传输和调度开销，最后把 **TTFT 控制在 0.5s 左右**。

这里有个关键点一定要主动补一句：

> 不是所有模块的时间简单相加，因为 RAG、用户特征、情绪识别是在并行跑，真正决定首包的是关键路径而不是总工时。

#### 4. 并发请求是怎么调度的
并发调度我会按“拆任务 + 定优先级 + 做资源隔离”来答。

**第一，拆成 DAG，而不是一条长串行链路。**

比如一次用户发言进来之后：

- ASR 增量片段出来后，立刻触发意图识别
- 在意图识别同时，并发触发：
  - RAG 检索
  - 用户特征加载
  - 情绪识别
- prompt 骨架先生成，不等所有增强结果都回来
- 哪些增强结果在预算时间内返回，就拼进去
- 超时没回来的直接降级，不阻塞 LLM 首包

这才叫调度。不是“开几个线程池”就算完了。

**第二，按优先级做调度。**

- **P0**：ASR、意图识别、主 prompt 组装、LLM 主响应
- **P1**：RAG、用户画像、情绪识别、实时推荐
- **P2**：埋点、分析、复盘

P0 任务一定有独立资源池和更高优先级；
P1 任务设 timeout budget，比如 80ms 或 100ms，超时直接放弃；
P2 完全异步，通过 MQ 或异步任务队列处理。

**第三，做资源隔离，避免互相拖死。**

工程上我们不会让所有任务共用一个线程池，而是至少做几层隔离：

- ASR pool
- RAG pool
- Tool / Workflow pool
- 异步事件消费 pool

否则高峰期只要 RAG 卡住，整个主链路也会一起抖。

#### 5. Prompt Cache / RAG Cache / 用户特征缓存是怎么做的
这个问题本质上不是“有没有 cache”，而是**哪些结果稳定、可复用、命中后能直接缩短关键路径**。

**（1）Prompt Cache**

我一般分两层讲：

- **前缀缓存（prefix cache）**
  - system prompt
  - few-shot
  - 固定业务规则
  - 这些内容变化很少，可以预组装，避免每次都重复拼 prompt

- **动态 prompt 缓存**
  - key 通常是：`query hash + 用户特征版本 + 场景标识`
  - value 可以是：
    - prompt 片段
    - query rewrite 结果
    - 某些稳定场景下的推荐话术模板

这里我不会吹“直接缓存 LLM 全量回答”，因为实时对话场景上下文强相关，命中率没那么高。更合理的是缓存 **可复用的中间结果**，比如 prompt 前缀、query rewrite、知识片段拼装结果。

**（2）RAG Cache**

- key：规范化 query 或 rewrite 后 query
- value：topK 检索结果 / rerank 结果
- TTL：通常短一些，防止知识更新后脏读太久

如果是高频问题场景，RAG cache 的价值很大，因为它能直接减少一次向量检索和重排时间。

**（3）用户特征缓存**

这个通常放 Redis：

- 用户画像
- 历史对话摘要
- 最近状态标签
- 常用业务字段

它的目标很明确，就是避免每轮对话都回源查数据库或重新计算用户状态。

#### 6. Streaming 是怎么优化体验的
这个一定要答对。**Streaming 本身不一定降低真实 latency，但它能显著降低用户感知延迟。**

我一般会这么说：

- **输入流式化**：ASR 是边说边转，不等用户整句说完才开始识别
- **输出流式化**：LLM token 一出来就往前端推，前端逐 token 渲染
- **结果分层输出**：先输出安全、通用、低风险的话术骨架，后续再补充 RAG 增强内容

这样即使完整回答要 1 秒多，用户在 300~500ms 内就能看到系统已经开始响应，体验会好很多。

一个很实用的工程点是：

> 我们不是等“最终完美答案”才返回，而是优先把“足够可用的首包”打出去。

#### 7. 如果 QPS 翻 10 倍，哪里先挂
这个问题一定不能答“FC 能弹性扩容，所以问题不大”。那是胡扯。

更真实的答案是：

**第一先挂的是 LLM 推理服务。**

因为它是最重的计算节点，尤其是 TTFT 对 GPU / 推理队列非常敏感。QPS 一上来，最先出现的是排队时间增长，TTFT 直接抖。

**第二个通常是 RAG 链路。**

包括：

- embedding 服务
- 向量检索
- rerank

这些模块吞吐上限都比普通应用逻辑更低，而且容易被热点 query 打穿。

**第三个是 Redis 和应用层资源池。**

Redis 在高并发下可能先撞到网络、CPU 或热点 key；
应用层如果线程池没隔离好，也会因为下游阻塞把工作线程耗尽。

**FC / 网关不一定第一个挂，但会暴露两个问题：**

- 冷启动带来的尾延迟
- 扩容速度赶不上下游真实承载能力

所以核心不是“能不能扩”，而是“扩出来之后下游吃不吃得下”。

#### 8. QPS x10 时怎么保系统不死
我会按“保护主链路”的思路答，而不是“所有能力都保住”。

**第一，限流。**

- 入口限流：按租户、按用户、按会话做配额控制
- 下游限流：RAG、Tool、外部服务单独做 token bucket / semaphore 控制
- 超预算直接拒绝或降级，而不是把系统拖死

**第二，资源隔离。**

- 主链路和增强链路分池
- 不让情绪识别、RAG 跟 LLM 主响应抢资源

**第三，降级。**

高并发下优先保：

- ASR
- 意图识别
- LLM 主响应

可以降级掉：

- 情绪识别
- 部分 RAG
- 实时推荐增强
- 非关键工具调用

最关键的一句话是：

> 在高并发下，我们追求的是“可用但不完美”，而不是“所有功能都在线但整体一起超时”。

**第四，缓存和预生成。**

- 高频 query 的 RAG 结果缓存
- prompt prefix cache
- 用户特征缓存
- 对确定性较高的下一步工具调用做预取或预生成

这些手段不是为了让系统“看起来聪明”，而是为了把重计算尽量前移。

#### 9. 项目经验（加分项）
在我们的实时外呼场景里，一个会话往往会同时触发 ASR、RAG、用户特征读取、情绪识别和实时话术推荐。如果把这些都串行执行，首包一定很差。

后来我们的做法是：

- 把主链路缩短成 `ASR -> 意图 -> 主 prompt -> LLM`
- 把 RAG、用户特征、情绪识别并发化
- 给增强任务加 timeout budget
- 命中缓存的直接复用
- 超时没回来的不等，先把首包发出去

这样做的结果是，TTFT 可以稳定控制在 0.5s 左右，而且并发上来时不会因为某个增强模块抖动，把整个系统一起拖死。

### 技术细节备忘（供追问参考）

**一、为什么 RAG 能并行？如果依赖 query rewrite 怎么办？**

这题是典型深挖。

核心回答是：

- 如果 query rewrite 是轻量级规则改写或模板改写，可以先快速做，再并行 RAG
- 如果 rewrite 本身依赖一个额外 LLM 调用，那就不能无脑放进首包关键路径

工程上更稳的做法通常是：

- 主链路先用原始 query 做一版快速检索
- rewrite 结果回来后，再触发增强检索或补充推荐

也就是说，**首包阶段追求的是“够快且够用”，而不是一次把检索做到最完美**。

**二、为什么不把所有结果都等齐再给 LLM？**

因为那样等于让最慢的那个模块决定整条链路。

在实时对话场景，正确做法是：

- 先满足主回答最小闭环
- 增强信息在预算内回来就用
- 超预算就降级

否则你不是在优化系统，你是在拿用户体验给“完美答案”陪葬。

**三、怎么监控延迟，而不是只看平均值？**

至少要看这些指标：

- TTFT P50 / P95 / P99
- ASR 延迟
- RAG 命中率和耗时
- LLM 首 token 耗时
- cache hit rate
- timeout / fallback / 降级比例

如果只看平均值，线上尾延迟会把你打得很难看。

**四、并发调度到底落在哪一层？**

可以这样答：

- **FC / 网关层**：承接入口流量、弹性扩容、做基础限流
- **应用编排层**：拆 DAG、做优先级调度、控制 timeout 和 join 策略
- **缓存层**：减少回源和重计算
- **MQ / 异步层**：把非实时任务彻底剥离

也就是说，调度不是一个点，而是一整套分层策略。

**五、面试里怎么收口？**

最后我一般会收成一句：

> 我们这套延迟优化的核心，不是单点把某个模块跑得更快，而是把首包路径控制住：关键路径做并行化，增强路径做预算和降级，非实时路径彻底异步化，这样系统在并发上来时还能稳住。
- 不必阻塞用户
- 高峰期容易堆积

这就是 MQ 的标准适用场景。把它同步做才奇怪。

**七、日志和 tracing 为什么走 MQ？**

因为日志、埋点、trace 事件数量大、写入频繁，而且对主流程来说通常不是强实时要求。先入 MQ 再异步处理，能显著减少主链路的 IO 压力，也更容易统一做清洗、聚合和检索。

**八、消费失败怎么办？**

我一般按三层处理：

- 短暂失败：自动重试
- 持续失败：进死信队列
- 最终失败：报警 + 人工补偿 / 任务重放

重点不是“永远不失败”，而是**失败后可观测、可重试、可补偿**。

**九、面试里怎么收口？**

可以直接用这句话收：

> 我们项目里用 MQ，核心不是因为它听起来高级，而是因为主链路必须足够轻，像通话后分析、日志观测、结果回写这种重任务就应该异步化。真正关键的不是把消息发出去，而是把不丢、幂等、顺序和失败补偿这几件事设计完整。
