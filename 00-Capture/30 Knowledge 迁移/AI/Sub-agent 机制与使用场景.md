# Sub-agent机制与使用场景

相关笔记：[[Skill的执行原理和渐进式披露设计理念]]

## 一句话理解

Sub-agent 不是“多开几个完整 Agent 一起聊天”，而是主代理在需要时，把某个子任务委托给一个**隔离上下文中的专用执行者**。  
它的本质是：

**主代理负责拆任务和汇总结果，子代理负责在干净上下文里完成单一子任务。**

---

## 1. 它解决的到底是什么问题

单个 Agent 在复杂任务里很容易遇到三个问题：

1. **上下文越来越脏**
   主任务、分支分析、临时探索、工具输出全混在一个上下文里，后续推理会越来越不稳定。

2. **不同任务需要不同角色**
   例如代码修改、事实核查、测试执行、文档整理，本质上是不同工作模式，放在一个 Agent 里容易互相干扰。

3. **主代理不应该沉到细节里**
   如果主代理既负责总体规划，又亲自处理每个局部细节，整体协调能力会快速下降。

所以 Sub-agent 的核心价值不是“显得高级”，而是：

- 做上下文隔离
- 做角色分工
- 做结果压缩

---

## 2. 核心模型：Orchestrator-Worker

Sub-agent 最适合用 **Orchestrator-Worker** 来理解。

### 主代理（Orchestrator）

主代理负责：

- 理解用户目标
- 拆分任务
- 判断是否需要委托
- 给子代理下达清晰任务
- 汇总多个子任务结果
- 决定最终输出

它更像项目经理或技术负责人，重点在于**协调**，而不是深入每个细节。

### 子代理（Worker）

子代理负责：

- 接收一个明确边界的任务
- 在自己的上下文中独立工作
- 使用必要工具完成探索或执行
- 返回压缩后的结论、结果或产物

它更像一个被临时拉来的专家，只对当前这一个子任务负责。

---

## 3. 它是怎么工作的

虽然表面上看像“多个 Agent 协作”，但底层更接近：

**主代理通过一个委托工具，创建一个新的隔离上下文，把任务交给子代理执行，再只拿回压缩结果。**

可以拆成 4 步。

### 3.1 任务匹配与委托

主代理先判断当前问题是否值得拆分，例如：

- 需要并行探索多个方向
- 某个子任务很重，可能污染主上下文
- 某个任务需要特定角色或特定 prompt

一旦决定委托，主代理会通过类似 `Task` / `Agent` 的工具把任务发出去。  
这个委托通常至少包含两部分：

- `description`：给人看的任务摘要
- `prompt`：真正交给子代理执行的详细任务说明

### 3.2 上下文分叉

系统会为子代理创建一个**全新的上下文窗口**。

这个上下文里通常只包含：

- 子代理自己的 system prompt
- 当前这次委托的任务说明
- 必要的环境信息
- 预加载的 skills 或工具权限配置

关键点在于：

**它不会自动继承主代理的完整历史上下文。**

这一步就是 Sub-agent 有效的根源。因为隔离本身，就是最彻底的“清理上下文”。

### 3.3 独立执行

子代理进入自己的工作流后，可以像一个小型 Agent 一样：

- 读文件
- 搜索信息
- 调用工具
- 进行多轮推理
- 生成中间结果

但它的目标非常单一：只完成这次被委托的任务。

### 3.4 结果压缩返回

子代理不会把完整过程原封不动地塞回主代理上下文。  
通常只会返回：

- 结论
- 摘要
- 关键发现
- 产物引用

主代理拿到的是**压缩后的结果**，不是完整的内部思考链路。

这意味着：

- 子代理可以在局部消耗很多 token
- 主代理上下文仍然保持相对干净

---

## 4. 为什么说它不是“真正的多 Agent 系统”

Sub-agent 虽然看起来像多代理协作，但通常并不是那种去中心化、自治式的 Agent 网络。

更准确地说，它是：

**单一主会话中的层级式委托机制。**

它和传统意义上的“多 Agent 系统”有几个关键差异。

### 4.1 不是平级协作，而是严格层级

主代理负责调度，子代理只向主代理汇报。  
子代理之间通常不直接通信。

### 4.2 没有共享记忆空间

每个子代理的上下文是隔离的。  
如果两个子代理需要共享信息，通常要通过主代理中转。

### 4.3 不一定是真正的系统级并行进程

它更像后端通过多次模型请求和上下文管理，模拟出多个“独立工作单元”。  
重点不在操作系统级别并发，而在**认知流程上的拆分与隔离**。

---

## 5. Main-agent 与 Sub-agent 的关键差异

这一部分是理解机制时最容易混淆的点。

| 维度 | Main-agent | Sub-agent |
|---|---|---|
| 角色 | 全局协调者 | 单点执行者 |
| 目标 | 理解整体目标、拆解、汇总、决策 | 完成某一个明确子任务 |
| 上下文 | 包含主任务历史，容易持续增长 | 新建、干净、边界明确 |
| Prompt | 更偏全局规则与通用能力 | 更偏角色约束与任务专注 |
| 输出 | 面向用户的最终答案 | 面向主代理的压缩结果 |
| 风险 | 容易被细节拖垮 | 容易因任务描述不清而跑偏 |

可以把它理解成：

- Main-agent 关心“整个项目怎么推进”
- Sub-agent 关心“我这一块具体怎么做完”

---

## 6. Prompt 管理为什么是核心

Sub-agent 强不强，很多时候不在于模型本身，而在于 **prompt 边界设计**。

因为它的本质不是“复制一个主代理”，而是创建一个**带专用 system prompt 的执行上下文**。

这里有三层关键设计。

### 6.1 静态定义

子代理可以预先定义自己的：

- 名称
- 描述
- system prompt
- 可用 tools
- 可加载 skills

这相当于先定义“这个专家是谁”。

### 6.2 运行时注入

主代理在委托时，再把这次任务的具体背景、限制条件、输出要求注入进去。

这相当于再补一句：

“你这次要解决的是这个具体问题，不是你的全部专业领域。”

### 6.3 利用隔离完成“删除”

Prompt 管理不只是加内容，也包括减内容。  
Sub-agent 最强的“减法”不是显式删除，而是：

**每次都从一个新上下文开始。**

这使它天然不会背上主代理过去那一大堆历史负担。

---

## 7. 落到 API 层，它实际长什么样

如果继续往下拆，Sub-agent 机制最终会落到标准的 LLM 消息协议上。  
以 Anthropic 为例，Claude Code/Claude Agent SDK 的底层核心就是 `/v1/messages`。

可以把它理解成：

- `system` 放持久规则
- `messages` 放会话历史
- `tools` 放当前允许调用的工具定义
- `tool_use` / `tool_result` 负责把“调用工具”和“工具返回”编码进对话流

一个简化后的消息体大致可以长这样：

```json
{
  "model": "claude-3-5-sonnet-20241022",
  "max_tokens": 8192,
  "system": "完整 system prompt",
  "messages": [
    {
      "role": "user",
      "content": [
        { "type": "text", "text": "用户输入或上一步总结" }
      ]
    },
    {
      "role": "assistant",
      "content": [
        { "type": "text", "text": "中间推理后的可见输出" },
        {
          "type": "tool_use",
          "id": "toolu_abc123",
          "name": "Task",
          "input": {
            "prompt": "交给子代理的详细任务"
          }
        }
      ]
    },
    {
      "role": "user",
      "content": [
        {
          "type": "tool_result",
          "tool_use_id": "toolu_abc123",
          "content": "子代理返回的压缩总结"
        }
      ]
    }
  ],
  "tools": [],
  "tool_choice": { "type": "auto" }
}
```

这里最关键的不是字段本身，而是这几个映射关系：

- 主代理对用户说话，本质上是在追加 `messages`
- 主代理调用子代理，本质上是在输出一个 `tool_use`
- 子代理执行完返回结果，本质上是在回填一个 `tool_result`
- 所谓“多代理编排”，在 API 层并不是神秘协议，而是**标准消息体 + 工具调用块**

---

## 8. Claude Code 如何把内部状态转成消息体

真正关键的不只是 API 格式，而是 Claude Code 内部维护的状态，如何在每一轮请求前被“编译”成最终消息体。

可以把它理解成：

**Claude Code 先维护一个内部会话状态，再在每次发请求前动态组装成 Messages API JSON。**

### 8.1 Main-agent 的组装逻辑

主代理并不是简单保存“所有原始消息”。  
它更像维护一个经过管理的内部状态，其中包括：

- 用户消息
- 可见的 assistant 消息
- 工具调用记录
- 工具返回结果
- CLAUDE.md 或同类持久规则
- 已发现但未展开的 skills 描述
- 已实际触发的 skill 内容

到真正发 API 请求前，系统才会做一次组装：

1. 把 CLAUDE.md、全局规则、环境说明拼进 `system`
2. 把 Skills 的 Level 1 信息作为“可发现能力”提供给主代理
3. 把当前仍保留的历史消息按顺序拼进 `messages`
4. 把当前允许的工具定义拼进 `tools`
5. 如果刚刚发生了工具调用结果，再把它编码成 `tool_result`

所以你看到的 JSON 请求体，更像是**内部状态的投影结果**，而不是 Claude Code 原生保存状态的样子。

### 8.2 Sub-agent 的组装逻辑

Sub-agent 更干净，也更容易理解。

当主代理决定委托时，后端通常会新建一个独立请求。  
这个请求的结构大体是：

- `system` = 子代理自己的 prompt + 预加载 skill 内容 + 少量环境信息
- `messages` = 一条 user message，也就是主代理交给它的任务书
- `tools` = 这个子代理被允许使用的工具

这意味着：

- 子代理不会继承主代理历史消息
- 子代理不会继承主代理完整 system prompt
- 子代理只知道“我是谁、我这次要干什么、我能用什么工具”

这就是为什么 Sub-agent 能保持专注。  
从消息体角度看，它不是“主代理上下文中的一个分支节点”，而是**一条重新发起的独立后端请求**。

### 8.3 Skill 与 Tool 在消息体里的位置

从实现角度看，Skill 和 Tool 都不是“魔法层”。

- Tool 会出现在 `tools` 定义中，并通过 `tool_use` / `tool_result` 参与消息流
- Skill 更像一种 prompt 资源，会在合适时机被注入 `system` 或变成后续消息的一部分

所以：

- Tool 解决“做什么动作”
- Skill 解决“按什么专业规则去做”

两者最终都会被折叠进一次具体 API 请求中。

## 9. 上下文是如何被删除和裁剪的

理解 Sub-agent，不能只看“如何添加上下文”，还要看“系统如何删上下文”。

因为只要系统长期运行，就一定会碰到 token 压力。  
这时上下文管理就会从“追加消息”变成“保留什么、丢弃什么、压缩什么”。

### 9.1 删除不是随机发生的，而是 token 压力驱动

系统在每轮请求前都会估算 token 占用。  
当占用接近窗口上限时，才会进入 pruning 或 compaction。

因此删除逻辑本质上不是：

“这条消息旧了就删。”

而是：

“在保持任务可继续的前提下，优先删除信息价值最低、但 token 占用最高的部分。”

### 9.2 最先被动刀的通常是旧工具输出

在实际系统里，最占 tokens 的往往不是用户问题，而是：

- 大段文件读取结果
- 长日志
- 搜索结果
- 测试输出
- 子代理返回的大段摘要

所以裁剪优先级通常是：

1. 旧的 `tool_result`
2. 旧的 `tool_use` 输入参数
3. 再往后才是较老的普通对话消息

这背后的原因很直接：

- 工具输出体积最大
- 其中很多只是过程性材料
- 相比删除用户目标，删除旧工具输出风险更低

### 9.3 常见裁剪手法不是“硬删”，而是先掩码、再压缩

为了避免一上来就丢信息，系统通常会分阶段处理：

- 先对旧工具结果做 masking，只保留“这里曾经有结果”的占位信息
- 再对更老的结果做 pruning，直接删掉正文
- 极端情况下再做 compaction，把长历史总结成更短的摘要

于是主代理上下文里最后留下的，往往不是原始材料，而是：

- 最近的完整上下文
- 较老结果的占位痕迹
- 更老阶段的压缩总结

### 9.4 Sub-agent 本身就是一种上下文经济手段

Sub-agent 的意义不只是分工，它本身就是一种非常有效的上下文节流机制。

因为它把大量局部探索放进一个独立窗口中处理，最后只把压缩后的结论带回主代理。  
换句话说：

- 如果没有 Sub-agent，主代理得直接吞下整个过程
- 有了 Sub-agent，主代理只需要吞下结果摘要

所以 Sub-agent 不只是“组织方式”，也是一种**context economy 策略**。

### 9.5 典型实现里，还会有分阶段阈值

在工程实现上，上下文裁剪往往不是“一刀切”，而是按 token 压力逐步升级。

常见思路是：

- 先告警，提示上下文压力升高
- 再对旧 `tool_result` 做 masking
- 再进一步 pruning，直接删除更老的工具输出
- 最后才进入 compaction，把长历史压成摘要

如果按你补充的那类实现细节来理解，可以把它看成一种近似分层：

- `70%` 左右：进入告警区
- `80%` 左右：开始 observation masking
- `85%` 左右：开始 fast pruning
- `90%` 左右：进入更激进的 masking
- `99%` 左右：进入 full compaction

这些具体百分比更像实现参数，而不是架构原理本身，版本之间可能调整。  
但它说明了一件很重要的事：

**上下文管理不是“满了再删”，而是持续监控、逐级退化、优先保住最关键语义。**

## 10. 适合什么场景

不是所有任务都值得启用 Sub-agent。  
它最适合的是那些**可以切出相对独立边界**的工作。

### 10.1 并行探索

例如：

- 同时研究多个方案
- 同时排查多个可能原因
- 同时阅读多个文件或模块

此时可以把不同方向交给不同子代理，再让主代理统一比较。

### 10.2 重型分析但不希望污染主上下文

例如：

- 深度代码审查
- 大量日志排查
- 长文档或多文件总结
- 大范围搜索与归纳

这类任务本身很耗上下文，交给子代理更合适。

### 10.3 需要明确角色切换

例如：

- 一个子代理专门做事实核查
- 一个子代理专门做测试执行
- 一个子代理专门做文档整理
- 一个子代理专门做代码修改建议

通过角色分工，可以减少“一会儿像分析师、一会儿像执行器”的混乱。

### 10.4 需要成本分层

如果系统支持不同模型配置，那么便宜模型可以承担大量局部工作，主代理只保留更高价值的全局决策。

---

## 11. 不适合什么场景

以下情况不一定值得引入 Sub-agent：

### 11.1 任务本身很短

如果一个问题一句话就能完成，委托成本反而高于直接处理。

### 11.2 子任务边界不清

如果主代理自己都没想清楚要子代理做什么，子代理只会把模糊问题放大。

### 11.3 高度依赖共享上下文

如果多个步骤强依赖同一段持续演化的上下文，而不能接受信息压缩损失，拆成子代理未必合适。

---

## 12. 它的真正收益与真正代价

### 收益

- 降低上下文污染
- 提高任务专注度
- 支持角色化分工
- 适合并行探索
- 有利于控制主代理上下文规模

### 代价

- 主代理必须会拆任务
- 委托描述必须足够清楚
- 子代理之间天然隔离，协作需要中转
- 结果压缩会带来信息损失
- 每次委托都有额外 token 和调度开销

所以 Sub-agent 不是“能力增强插件”，而是一个**用工程化方式管理上下文和认知负载的机制**。

---

## 13. 一个更准确的心智模型

如果只记一个结论，可以记这个：

> Sub-agent = 用新的上下文窗口，临时创建一个专注处理局部问题的执行者；  
> 主代理负责派活、收结果、做最终判断。

因此它的关键不在“有几个 Agent”，而在三个动作：

1. **拆**
2. **隔离**
3. **压缩回收**

只要这三个动作成立，Sub-agent 就有价值；  
如果拆不清、隔离无收益、压缩后信息反而不够用，那它就只是形式上的复杂化。

---

## 14. 总结

Sub-agent 的本质不是炫技式多代理，而是一种围绕上下文管理建立起来的任务委托机制。

它背后的逻辑很简单：

- 主代理保留全局视角
- 子代理承担局部重活
- 通过上下文隔离避免污染
- 通过结果压缩把局部工作重新接回全局决策

从这个角度看，Sub-agent 更像一种 **LLM 时代的任务编排模式**，而不是传统软件意义上的进程管理机制。

- `Sub-agent 的 prompt 设计方法`

- `什么任务值得拆给 Sub-agent，什么不值得`
