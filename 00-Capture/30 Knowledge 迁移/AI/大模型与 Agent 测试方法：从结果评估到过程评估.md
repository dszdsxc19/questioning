# 大模型与 Agent 测试方法：从结果评估到过程评估

很多团队一开始做大模型应用时，测试方式都很朴素：挑几个问题问一问，看答案像不像对，再看看排行榜选个分数高的模型。

但一旦系统从单轮问答走向 RAG、Tool Calling、Workflow、Multi-Agent，这种测试方式就很快不够用了。因为你真正要验证的，已经不只是“最后一句话对不对”，而是：

- 任务有没有完成
- 工具有没有选对、用对
- 中间步骤是不是合理
- 换 prompt、换模型、换 agent 配置之后，到底有没有整体变好
- 线上真实流量里，质量有没有悄悄回退

这也是为什么今天的评测，正在从 **模型测试** 走向 **Agent 测试**，从 **只关注结果** 走向 **同时关注过程**，再进一步走向 **A/B 实验和生产监控**。

---

## 一、为什么“测模型”会演进成“测 Agent”

传统大模型测试更像在测一个“回答器”。

你给它一道题，它给你一个答案，然后用标准答案、模型裁判或人工偏好去打分。这一层依然重要，例如：

- 回答是否相关
- 是否忠于上下文
- 有没有幻觉
- 总体回答质量如何

但 Agent 的本质不是只“回答”，而是“为了完成任务而采取一系列动作”。一旦系统开始调用搜索、数据库、工作流、外部 API、浏览器工具，你就会发现只测最终答案有三个明显问题：

### 1. 最终结果正确，不代表过程健康

Agent 可能绕了很多无效步骤，甚至误调了几次工具，最后靠运气把答案拼对了。这样的系统一上线，成本、延迟和稳定性都会出问题。

### 2. 最终结果错误，但你不知道错在哪一层

是模型理解错了任务？检索错了资料？调用了错误的工具？工具参数不对？还是规划路径有冗余？如果没有过程数据，排查几乎只能靠猜。

### 3. 改动越来越频繁，人工体验越来越不可靠

Prompt 改一点、模型换一版、工具 schema 调一下，系统行为就可能变化。靠“我试了几个例子，感觉还行”已经支撑不了持续迭代。

所以，Agent 测试的核心变化不是多了几个工具名字，而是**测试对象变了**：

- 从“单次回答质量”变成“任务执行系统质量”
- 从“单点打分”变成“分层评估”
- 从“上线前看一下”变成“离线实验 + 在线监控”

## 二、今天主流的四层测试框架

如果你只记一个框架，我建议记下面这四层：

1. **结果评估（Outcome / Black-box）**
2. **过程评估（Trajectory / Glass-box）**
3. **对比实验（Offline / Online A/B）**
4. **生产反馈（Live Evals / Human Feedback / Monitoring）**

它们不是替代关系，而是递进关系。

### 1. 结果评估：先判断“任务有没有做成”

这是最基础的一层，也是大多数团队最先做的一层。

你把 Agent 当成黑盒，只看输入和最终输出，典型问题包括：

- 最终回答是否正确
- 是否完成任务
- 是否忠于检索上下文
- 是否出现幻觉
- 是否满足格式、风格或业务规则

这一层适合回答的问题是：

> “如果我只看最终交付结果，这个系统到底好不好用？”

典型指标包括：

- `Answer Relevancy`
- `Faithfulness`
- `Hallucination`
- `Task Completion`
- 自定义 rubric 或 LLM-as-a-Judge

这层评估非常适合：

- RAG 问答
- 摘要、分类、抽取
- 单轮或短链路 Agent
- 回归测试和 CI

但它的局限也很明显：**只能告诉你结果怎样，不一定告诉你为什么。**

### 2. 过程评估：再判断“它是怎么做到的”

当系统进入 Agent 阶段，过程评估的重要性会迅速上升。

这里关注的是执行轨迹，也就是 trace / trajectory / step sequence，而不是只看最后一句话。典型问题包括：

- 工具有没有调用正确
- 工具调用顺序是否合理
- 有没有多余步骤
- 是否在错误的地方循环
- 规划是否稳定
- 哪一步导致了最终失败

这一层适合回答的问题是：

> “这个 Agent 的做事方式对不对？”

过程评估常见的几种视角：

- **工具正确性**：该调什么工具，参数对不对，顺序对不对
- **轨迹匹配**：实际轨迹与参考轨迹是否一致
- **步骤级评分**：对单个 step、span、tool call 分别打分
- **效率视角**：有没有无效调用、重复检索、过长链路

这层的价值非常大，因为它把“黑盒猜测”变成“白盒定位”。

### 3. A/B 测试：比较不同变体，而不是凭感觉改

很多人会把 A/B 测试理解成“线上分流实验”，但在 LLM/Agent 领域里，它更广义地表示：

> **系统性比较多个 prompt / 模型 / 配置 / agent 版本的实验。**

绝大多数团队最常做的是 **离线 A/B**：

- 固定一份 dataset
- 跑多个模型或 prompt 版本
- 比较结果指标和过程指标
- 选择胜出的版本继续推进

常见对比对象包括：

- `GPT-5` vs `Claude` vs `Gemini`
- 同一模型的不同 system prompt
- 不同检索策略
- 不同工具集配置
- 单 Agent vs 多 Agent

只有当离线结果足够稳定后，才有必要做 **在线 A/B**，例如：

- shadow 流量
- canary 分流
- 小流量试点

LLM 系统里最常见的错误不是“不做线上实验”，而是**连离线对比都没做就直接上线**。

### 4. 生产反馈：让评测从一次性动作变成持续闭环

真正成熟的团队，不会把评测理解成“开发阶段跑一下脚本”。

他们会把线上真实流量继续纳入评测闭环，例如：

- 对生产 trace 做采样评分
- 对关键 step 跑在线 eval
- 收集用户反馈
- 从失败案例反向构造 dataset
- 监控质量、成本、延迟是否一起恶化

这是因为 Agent 的退化，常常不是突然完全坏掉，而是：

- 工具调用成功率慢慢下降
- 某类任务开始频繁绕远路
- 新 prompt 让一部分场景变好，但另一部分场景变差
- 成本和延迟显著上升，但最终回答表面上还凑合

如果没有生产反馈，很多问题只有等业务投诉时才会暴露。

## 三、从“关注结果”到“关注过程”，是 Agent 时代最关键的变化

如果说过去的大模型评测重心是“这句话像不像对”，那今天 Agent 评测的重心更像是：

> “它有没有以一种可靠、可解释、可复现的方式把事做成。”

这会带来三个非常现实的变化。

### 1. 评测对象从 answer 变成 trace

过去一条样本大概是：

`input -> output`

现在更像是：

`input -> planning -> tool calls -> intermediate outputs -> final output`

这意味着 dataset、评测指标、调试界面都要升级。

### 2. 评测指标从“正确率”扩展到“执行质量”

Agent 系统常见的失败，不只是答错，还包括：

- 做了不该做的动作
- 漏了必须做的动作
- 走了太长路径
- 在错误节点反复重试
- 工具和模型各自都没错，但组合起来出错

所以现在的指标体系会同时看：

- 结果对不对
- 工具用得对不对
- 路径走得对不对
- 成本和延迟是否可接受

### 3. 调试方式从“读输出”变成“读轨迹”

做 Agent，trace 几乎已经不是可选项，而是基础设施。

没有 trace：

- 你无法解释失败原因
- 无法构造高质量 dataset
- 无法做步骤级评分
- 无法把线上问题回灌到离线实验

所以“评测”和“可观测性”正在越来越紧地绑在一起。

## 四、工具地图：这些工具分别适合什么

截至 **2026-04-07**，这一领域常见工具大致可以这样理解：

| 方向 | 代表工具 | 更适合解决什么问题 |
| --- | --- | --- |
| 结果评估 | DeepEval / Confident AI | 用一组指标系统性评估最终结果、任务完成度与安全性 |
| 过程评估 | Langfuse / LangSmith | 通过 trace、trajectory、online eval 看 Agent 的执行过程 |
| Prompt / Model 对比 | Promptfoo | 快速比较不同 prompt、模型、provider 或配置 |
| 框架原生评测 | Mastra Scorers | 在 Mastra 体系内直接给 agent / workflow 加评分能力 |
| 统一观测底座 | OpenTelemetry + 各类后端 | 把 trace、成本、延迟、错误和业务标签统一起来 |

下面只讲定位差异，不展开具体 API。

### DeepEval / Confident AI：最像“LLM 测试框架”

如果你主要关心的是**最终效果评估**，DeepEval 依然是非常强的一类方案。

它适合做的事情包括：

- 对最终输出跑质量指标
- 对 agent 跑 `Task Completion`
- 对工具调用跑 `Tool Correctness`
- 在 CI/CD 里做回归测试

它的优势是：

- 指标体系完整
- 非常适合把“感觉好像不错”变成可重复的测试
- 能把结果评估写进工程流程

它的边界也很清楚：

- 更偏“评估引擎”
- 过程洞察要依赖 tracing 能力配合

### Langfuse：过程观测与实验闭环很强

Langfuse 的强项不只是 trace，而是把 **trace、dataset、experiment、prompt version、production eval** 串成一个闭环。

它适合的场景包括：

- 看 Agent 轨迹和步骤细节
- 从真实 trace 回收数据集
- 在 dataset 上比较不同版本
- 追踪 prompt / model / version 的线上表现

如果你的问题是：

> “这个 Agent 到底在哪一步开始变差的？”

Langfuse 往往比单纯的结果打分工具更有帮助。

### LangSmith：适合做 trajectory eval 和开发期对比

LangSmith 的优势在于：

- tracing 完整
- offline / online eval 都成熟
- trajectory match 很适合有参考路径的 agent
- experiment compare 适合版本对比

如果你的 Agent 工作流比较明确，且你希望验证“这条路径是不是按预期执行”，LangSmith 会很顺手。

### Promptfoo：最轻量的对比实验工具之一

Promptfoo 的长处不是全栈 observability，而是**快速做对比实验**。

它很适合：

- prompt 迭代
- 模型横向对比
- provider 切换对比
- 基本断言验证
- 红队和安全测试

如果你想先快速回答这个问题：

> “同一组测试集上，哪一个 prompt / 哪一个模型更稳？”

Promptfoo 往往是启动成本很低的选择。

### Mastra Scorers：适合已经重度使用 Mastra 的团队

如果你本来就在 Mastra 体系内开发 agent，那么原生 scorers / evals 的体验会很自然。

它适合：

- 直接在 agent 或 workflow 上挂评测
- 用内置 scorer 验证回答质量
- 在 Mastra 的 observability 体系里一起看结果

它的优势不是“覆盖所有评测场景”，而是**在 Mastra 框架内足够顺手、集成成本低**。

## 五、A/B 测试到底怎么理解，才不容易走偏

LLM/Agent 领域里，A/B 测试最容易被误解的有两点。

### 误解一：A/B 测试一定是线上分流

不是。

在线上分流之前，绝大多数团队先做的是 **离线 A/B**。也就是在固定数据集上，比较不同变体的结果。

这是因为线上 A/B 成本更高，风险也更高。离线都没有明显优势的版本，没有必要直接碰真实用户流量。

### 误解二：A/B 只是在比 prompt

也不是。

你真正可以拿来做 A/B 的，至少包括：

- 模型
- prompt
- tool schema
- 检索策略
- routing 策略
- memory 策略
- 单 Agent / 多 Agent 架构

本质上，A/B 是一种**实验方法**，不是某个特定功能。

### 一个更实用的理解

你可以把它拆成两步：

1. **离线 A/B**：在固定 dataset 上比结果和过程
2. **在线 A/B**：在真实流量上验证业务价值、成本和稳定性

只有两步都建立起来，系统才算真正进入可控迭代。

## 六、一个最小可落地的评测闭环

如果你现在还没有完整体系，不需要一上来就把所有工具都接满。

更现实的做法是按下面这个顺序搭：

### 第一步：先准备一份小而真的 dataset

不要先追求几千条样本。先收集 20 到 50 条最能代表真实问题的数据：

- 高频任务
- 容易失败的边界 case
- 用户投诉过的 case
- 高价值任务

如果能从生产 trace 反向抽样，通常比拍脑袋造题更有价值。

### 第二步：先定义结果指标

至少先有一组最基本的 outcome 指标，例如：

- 任务完成
- 幻觉
- 忠实度
- 格式正确性

这一步的目标不是完美，而是先把“看感觉”替换成“可重复测”。

### 第三步：把 trace 打起来，再补过程指标

当你的系统已经有 Tool Calling、RAG、Workflow 时，就该尽快加上 trace，并开始关注：

- 工具调用准确性
- 轨迹是否合理
- 是否有无效步骤
- 哪一步最常出错

### 第四步：把改动都变成离线 A/B

以后每次改 prompt、换模型、调工具，不要直接凭体验决定，而是：

- 跑同一份数据集
- 比较结果指标
- 对照过程轨迹
- 记录回归和提升

### 第五步：把线上反馈再回灌回来

把线上真实失败 case 回灌到 dataset 里，形成一个持续增长的测试集。

这样你的评测才不是一次性脚本，而是一个会越来越贴近业务的系统。

## 七、几个常见误区

### 1. 只看 benchmark，不看自己的任务

排行榜可以帮你筛选候选模型，但不能替你做最终选择。真实业务数据才是最后的判官。关于这点，可以结合 [[大模型评估指南：如何看懂榜单并选择适合自己的模型]] 一起看。

### 2. 只看最终答案，不看轨迹

这会让你在 Agent 时代失去最关键的定位能力。过程可见性是 Agent 工程的基础设施，不是锦上添花。相关背景可参考 [[AI Tracing 语义规范与 Mastra 实现]]。

### 3. 只做离线评测，不看线上表现

离线测试集永远不可能覆盖全部真实流量。生产反馈、用户反馈、trace 采样和在线评估是必要补充。

### 4. 只接工具，不定义评测问题

先问“我要验证什么”，再选工具。否则很容易陷入工具齐全，但没有判断标准的假繁荣。

### 5. 只追求质量，不看成本和延迟

Agent 系统的“变好”不是单一维度。一个稍微更准、但成本翻倍、延迟翻倍的版本，未必是真正更好的版本。

---

## 八、总结

今天的大模型与 Agent 测试，已经不再是“跑几个例子看看感觉”。

更准确地说，它正在变成一套完整工程方法：

- **结果评估**：任务有没有做成
- **过程评估**：它是怎么做成的
- **A/B 测试**：不同版本到底谁更好
- **生产反馈**：线上质量有没有持续稳定

如果只做第一层，你能知道系统“好不好”；如果把四层都建立起来，你才真正开始知道系统“为什么会好、为什么会坏，以及该怎么稳定迭代”。

对今天的 Agent 团队来说，这种能力已经不再是加分项，而是基础能力。
