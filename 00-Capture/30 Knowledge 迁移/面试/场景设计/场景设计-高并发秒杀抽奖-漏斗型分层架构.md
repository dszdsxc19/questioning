## 场景描述

**题目**：设计一个抽奖系统，每人可抽 m 次，奖品总数共 n 个，需要支持百万级高并发。

---

## 核心难点

| 难点        | 说明                       |
| --------- | ------------------------ |
| **高并发读写** | 百万级 QPS，系统压力巨大           |
| **防超发**   | 奖品总数 n 必须精准控制，并发扣减极易出现负数 |
| **资格限制**  | 每人限制 m 次，需要高性能计数器防刷      |
|           |                          |

> 💡 **核心思想**：采用**"漏斗型"分层拦截架构**，尽量把请求拦截在系统上游，层层过滤，最后只有极少数真实中奖的流量落入数据库。

## 一、接入层：流量拦截与防抖

### 1. 静态资源 CDN 缓存

```
HTML、CSS、JS、图片 ──推送──> CDN 边缘节点 ──> 用户就近获取
```

**效果**：极大降低源站压力

### 2. 前端拦截

| 措施 | 说明 |
|-----|------|
| 按钮置灰 | 点击后立即禁用按钮，防止连击 |
| 倒计时保护 | 开启短时间倒计时（如 1-2 秒） |
| 防抖处理 | 节流函数控制请求频率 |

### 3. 网关限流

#### 限流算法对比

| 算法 | 原理 | 特点 |
|-----|------|------|
| **漏桶** | 以恒定速率处理请求，多余流量溢出 | 强行平滑流量，无突发能力 |
| **令牌桶** | 以恒定速率放入令牌，请求需获取令牌 | 允许一定突发流量，更灵活 |

> 💡 详见：[[限流算法-令牌桶与漏桶]] → 详细原理、代码实现、参数配置
>
> 💡 **推荐**：令牌桶算法更适合互联网场景，允许应对一定程度的突发流量。

**限流策略**：
- 全局限流：基于令牌桶配置系统阈值
- 用户限流：拦截恶意 IP、黑产账号
- 风控拦截：识别异常行为模式

## 二、缓存层：核心业务逻辑前置

高并发下的抽奖逻辑必须全部在 Redis 中完成。

### 1. 用户次数扣减（防刷）

利用 Redis 单线程特性：

```bash
# 每次 INCR，配合过期时间
INCR user:123:draw_count
EXPIRE user:123:draw_count 3600

# 返回值 > m 则拦截
```

### 2. 奖品库存原子扣减（防超发）

**问题**：并发情况下，"查询库存"和"扣减库存"必须原子执行。

**解决**：使用 Redis Lua 脚本，Redis 执行 Lua 时具有单线程排他性。

#### 核心 Lua 脚本

```lua
local key = KEYS[1]
local stock = tonumber(redis.call('get', key))

if (stock == nil) then
    return 0  -- 活动未初始化
end

if (stock > 0) then
    redis.call('decr', key)
    return 1   -- 扣减成功（中奖）
else
    return -1  -- 库存不足（未中奖）
end
```

**调用逻辑**：

| 返回值 | 含义 | 后续处理 |
|-------|------|---------|
| `0` | 活动未初始化 | 提示活动未开始 |
| `1` | 库存扣减成功 | 进入抽奖算法 |
| `-1` | 库存已耗尽 | 直接返回未中奖 |

## 三、应用层：抽奖算法

经过网关限流、次数校验、库存校验后，真正进入应用层的流量已大幅减少。

```javascript
// 伪代码示例
function draw(userId) {
    // 1. 校验次数
    const count = redis.incr(`user:${userId}:count`);
    if (count > m) return { code: 40001, msg: "次数已用尽" };

    // 2. 扣减库存
    const result = redis.eval(luaScript, ["prize:stock"]);
    if (result !== 1) return { code: 40002, msg: "奖品已抢完" };

    // 3. 执行抽奖算法
    const prizes = loadProbabilityConfig();
    const winner = randomDraw(prizes);

    if (winner) {
        // 4. 中奖：投递消息队列
        mq.send({ userId, prize: winner.id, timestamp: Date.now() });
        return { code: 200, data: { prize: winner.name } };
    } else {
        return { code: 200, data: { prize: null } };
    }
}
```

## 四、消息队列：异步削峰落库

**绝对不能同步调用 MySQL**，会严重拖慢接口响应时间。

### 流程设计

```
应用层 ──投递消息──> Kafka/RocketMQ ──消费──> MySQL
        │                          │
        └── 立即响应用户          └── 平稳速率写入
```

**优势**：
- ✅ 接口响应极快
- ✅ 削峰填谷，保护数据库
- ✅ 消息队列保证不丢失

## 五、持久化层：最终一致性

### 消费端设计

```
while (true) {
    message = mq.poll();

    // 幂等性保障
    if (isProcessed(message.id)) continue;

    // 执行数据库操作
    transaction {
        1. 扣减真实库存
        2. 生成发奖凭证
        3. 更新用户资产
    }

    markProcessed(message.id);
}
```

### 幂等性实现

| 方案 | 说明 |
|-----|------|
| 唯一索引 | 数据库表加唯一索引，重复插入报错忽略 |
| 防重表 | 专门的去重表记录已处理的消息 ID |
| 状态机 | 基于状态流转，重复操作直接跳过 |

## 六、兜底与容灾

| 组件 | 容灾措施 |
|-----|---------|
| **Redis** | Cluster 集群，防止单点故障 |
| **消息队列** | 消息持久化，消费端幂等 |

## 七、前端优雅降级

### 痛点分析

直接返回 HTTP 429 会导致：
- 触发底层网络异常拦截器
- 显示冷冰冰的 "Network Error"
- 引起用户恐慌

### 最佳实践方案

#### 1. 网关层重写响应

```json
HTTP/1.1 200 OK
Content-Type: application/json

{
  "code": 40029,
  "msg": "活动太火爆啦，请稍后再试",
  "data": null
}
```

#### 2. 响应头建议

```
Retry-After: 3  // 建议等待 3 秒后重试
```

#### 3. 前端降级逻辑

```javascript
// Axios 拦截器
axios.interceptors.response.use(
    response => {
        const { code } = response.data;

        if (code === 40029) {
            // 显示温馨提示
            showToast("活动太火爆啦，请稍后再试~");

            // 按钮置灰 + 倒计时
            disableButton(3);  // 读取 Retry-After 或默认值

            // 排队动画（可选）
            showQueueAnimation();
        }

        return response;
    }
);
```

### 限流响应类型总结

| 类型 | HTTP 状态码 | 说明 |
|-----|------------|------|
| 标准协议 | `429 Too Many Requests` | 标准 HTTP 限流状态码 |
| 旧系统 | `503 Service Unavailable` | 部分老中间件使用 |
| **业务封装** | `200 OK + 业务码` | **推荐，体验最佳** |

## 面试要点总结

### 1. 为什么要分层拦截？

- **层层过滤**，每层拦截不同类型的无效请求
- **保护下游**，避免核心服务被打垮
- **用户体验**，快速拒绝，不给用户虚假希望

### 2. 为什么使用 Lua 脚本？

- Redis 执行 Lua 时**单线程排他**
- 保证"查询 + 扣减"的**原子性**
- 避免 **ABA 问题**和**超发**

### 3. 为什么要消息队列？

- **异步解耦**：接口响应不依赖数据库写入
- **削峰填谷**：平滑流量，保护数据库
- **最终一致性**：通过重试和幂等保证数据准确

### 4. 漏桶 vs 令牌桶怎么选？

| 场景 | 推荐算法 |
|-----|---------|
| 严格恒定速率 | 漏桶 |
| 允许一定突发 | **令牌桶**（推荐） |

### 5. Redis 为什么也会崩溃？

- 单节点 QPS 通常在 **十万级别**
- 百万流量会打满**网卡带宽**
- 需要分层拦截，不能全靠 Redis
