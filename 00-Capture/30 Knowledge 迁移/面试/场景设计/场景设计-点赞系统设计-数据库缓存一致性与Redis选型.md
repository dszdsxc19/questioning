# 点赞系统设计 - 数据库、缓存一致性与 Redis 选型

## 面试官想听到的核心答案

如果让我设计点赞系统，我不会一上来就说 Redis，很容易答成八股。这个题先看数据，再看压力。

点赞场景里真正的核心数据只有四类：**点赞关系、点赞总数、帖子热度、点赞变更事件**。冷帖并发低，直接写 MySQL 就够了；热点帖子再进 Redis，把高并发压力挡在缓存层。点赞关系放 `Set` 去重，点赞计数和热度指标单独维护，数据库负责持久化和冷数据兜底，Redis 只做热点加速层。写路径上可以用 MQ 或定时任务异步落库，接受最终一致性；Redis 故障时直接降级回数据库，保证服务不崩。

一句话总结：**冷数据直写 DB，热点数据进 Redis，缓存扛并发，数据库保事实，异步链路做持久化，降级方案保可用性。**

---

## 一、先看数据模型

坏回答先背组件。好回答先拆数据。

### 1. 核心数据有哪些

| 数据 | 作用 | 典型存储 |
|------|------|----------|
| `post_like_rel(post_id, user_id)` | 表示谁给哪篇帖子点过赞 | MySQL / Redis Set |
| `post.like_count` | 帖子的点赞总数 | MySQL / Redis String |
| `post.hot_score`、`view_count`、`last_active_at` | 判断是否值得进缓存 | MySQL / Redis String / ZSet |
| `like_event(post_id, user_id, action)` | 异步持久化和回放修正 | MQ / Binlog / 任务表 |

### 2. 数据关系怎么理解

- **点赞关系**是真正的事实表，决定“这个用户到底点没点”。
- **点赞总数**是派生数据，本质上可以从关系表聚合出来，但高并发下不能每次都现算。
- **帖子热度**不是点赞关系本身，而是资源调度信号，用来决定要不要进缓存。
- **点赞事件**不是给用户看的，而是给系统兜底和解耦用的。

### 3. 数据库表怎么建

最小可用版本就两张表够了：

```sql
CREATE TABLE post (
  id BIGINT PRIMARY KEY,
  like_count BIGINT NOT NULL DEFAULT 0,
  view_count BIGINT NOT NULL DEFAULT 0,
  hot_score BIGINT NOT NULL DEFAULT 0,
  created_at DATETIME NOT NULL,
  updated_at DATETIME NOT NULL
);

CREATE TABLE post_like_rel (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  post_id BIGINT NOT NULL,
  user_id BIGINT NOT NULL,
  status TINYINT NOT NULL DEFAULT 1,
  created_at DATETIME NOT NULL,
  updated_at DATETIME NOT NULL,
  UNIQUE KEY uk_post_user (post_id, user_id),
  KEY idx_user_post (user_id, post_id)
);
```

这里的关键不是字段多漂亮，而是 **`(post_id, user_id)` 必须唯一**。不然你前面讲再多缓存，底层事实都不可靠。

## 二、冷热分层与缓存触发

### 1. 冷帖为什么不该默认进缓存

很多人一讲高并发就想“所有帖子都放 Redis”。这就是典型的资源浪费。

新帖子、冷门帖子、低访问帖子，并发压力通常很低。对这些帖子来说：

- 点赞直接写数据库
- 查询点赞状态直接查数据库
- 点赞数直接读数据库字段

这样做的好处很朴素：

- 不浪费 Redis 内存
- 不需要维护大量低价值 Key
- 减少缓存预热、淘汰、修复复杂度

### 2. 热帖什么时候进缓存

当帖子满足任一条件时，可以把它升级为热帖：

- 点赞数超过阈值，例如 `>= 100`
- 访问量超过阈值，例如 `>= 1000`
- 最近 5 分钟内交互频次明显升高

这里的阈值不是业务真理，只是工程手段。目的是把**真正有并发压力的对象**筛出来。

### 3. 热帖怎么加载到 Redis

常见两种触发方式：

#### 实时触发

- 用户点赞或访问时检查热度
- 达标后立刻从 MySQL 拉取点赞用户集合
- 写入 Redis 并设置 TTL

优点是响应快，缺点是瞬时高峰下容易把加载动作打集中。

#### 定时扫描

- 后台任务按分钟扫描热点帖子
- 批量预热 Redis

优点是温和，缺点是热度变化反应稍慢。

现实里可以混用：**定时扫描负责主预热，实时触发负责补漏。**

### 4. 一个热帖需要哪些 Redis Key

```text
like:post:{postId}                 -> Set，存点赞用户 ID
like:count:post:{postId}           -> String，存实时点赞数
like:meta:post:{postId}            -> Hash，存热度元信息
hot:post:ranking                   -> ZSet，存热帖排序
```

这一步很重要。不要把所有职责都塞进一个 Key，不然以后只会越来越丑。

## 三、Redis 写路径与读路径

### 1. 点赞关系为什么用 Set

因为点赞天然是“无序去重集合”：

- 一个用户对同一帖子只能点一次
- 需要快速判断是否已点赞
- 需要快速增删用户关系

这正是 `Set` 的职责，不需要发明新轮子。

| 操作 | Redis 命令 | 说明 |
|------|------------|------|
| 点赞 | `SADD like:post:{postId} {userId}` | 新点赞返回 `1`，重复点赞返回 `0` |
| 取消点赞 | `SREM like:post:{postId} {userId}` | 删除点赞关系 |
| 是否已点赞 | `SISMEMBER like:post:{postId} {userId}` | O(1) 判断 |
| 点赞人数 | `SCARD like:post:{postId}` | 获取集合大小 |

### 2. 写路径怎么走

#### 冷帖写路径

```text
用户点赞 -> 应用服务 -> MySQL 插入/更新 -> 返回结果
```

这里靠数据库唯一索引兜底：

- 插入成功，说明首次点赞
- 唯一键冲突，说明重复点赞

#### 热帖写路径

```text
用户点赞
  -> Redis SADD
  -> 成功则更新实时计数/热度
  -> 发送 like_event
  -> 立即返回前端
```

伪代码：

```python
def like_post(post_id, user_id):
    if not is_hot_post(post_id):
        return like_post_in_db(post_id, user_id)

    added = redis.sadd(f"like:post:{post_id}", user_id)
    if added == 0:
        return {"liked": True, "msg": "already liked"}

    redis.incr(f"like:count:post:{post_id}")
    redis.hincrby(f"like:meta:post:{post_id}", "ops_5m", 1)
    mq.publish("like_event", {
        "post_id": post_id,
        "user_id": user_id,
        "action": "like"
    })
    refresh_ttl(post_id)
    return {"liked": True}
```

### 3. 取消点赞也要讲清楚

很多人只会讲点赞，不讲取消，这就是半套设计。

```text
用户取消点赞
  -> Redis SREM
  -> 成功则递减实时计数
  -> 发送 unlike_event
  -> 异步落库
```

### 4. 查询路径怎么走

#### 查询“我是否点过赞”

- 热帖：`SISMEMBER`
- 冷帖：查 `post_like_rel`

#### 查询“帖子有多少赞”

- 热帖：优先读 Redis 实时计数
- 冷帖：读 MySQL 的 `like_count`

注意这里的好品味：**不要把 `SCARD` 当唯一真相。**

因为：

- 大集合频繁 `SCARD` 虽然可用，但你迟早会想做更多派生统计
- 独立维护 `like_count` 更方便聚合和回源
- 数据修正时也更容易做双向校验

## 四、异步持久化与最终一致性

Redis 很快，但它不该是你的最终事实源。事实还是要落 MySQL。

### 1. 方案一：定时批量同步

流程：

1. 定时任务扫描热点帖子
2. 对比 Redis 与 MySQL 的差异
3. 批量补写新增点赞和取消点赞
4. 顺便修正 `like_count`

优点：

- 实现简单
- 对业务代码侵入小
- 适合中小规模系统

缺点：

- 数据延迟更明显
- Redis 宕机时可能丢失最近一段时间变更

### 2. 方案二：MQ 事件驱动同步

流程：

1. Redis 写成功后发送 `like_event`
2. 消费者异步写入 `post_like_rel`
3. 同步更新 `post.like_count`
4. 失败则重试或进入死信队列

优点：

- 实时性更强
- 削峰填谷
- 写链路职责更清晰

缺点：

- 引入消息可靠性、重复消费、积压处理等额外复杂度

### 3. 面试时怎么做取舍

如果是中小系统，可以先答：

> 我会先用定时任务同步，因为实现简单，能满足业务。

如果题目明确是大流量社交平台，再升级成：

> 热点帖子进 Redis，变更投递 MQ，消费者异步落库，接受最终一致性。

别上来就 Kafka、分库分表、全链路 CDC，那不是设计，是表演。

### 4. 一致性怎么回答

点赞场景通常不要求强一致。用户能接受短时间内点赞数有轻微延迟，但不能接受接口很慢或经常失败。

所以默认取舍应当是：

- **用户体验优先**
- **系统吞吐优先**
- **数据最终一致**

只有在题目明确要求“金融级强一致”时，才考虑“先写 DB，再删缓存/回填缓存”那套更重的路径。但对点赞来说，这通常是过度设计。

## 五、缓存淘汰、降级与容灾

### 1. TTL 自动淘汰

给热点 Key 设置过期时间，例如 1 天：

```text
EXPIRE like:post:{postId} 86400
EXPIRE like:count:post:{postId} 86400
```

每次访问或点赞时刷新 TTL。这样只要帖子还有热度，缓存就继续活；一旦冷下来，就自动释放内存。

### 2. 主动淘汰

后台任务扫描 Redis 元数据：

- 最近交互低
- 点赞数低于热帖阈值
- 长时间无访问

满足条件就主动删 Key。

这比单靠 TTL 更可控，适合对内存占用更敏感的系统。

### 3. Redis 内存淘汰策略

配置 Redis `maxmemory-policy` 做兜底，比如：

- `allkeys-lru`
- `allkeys-lfu`

这不是业务策略，只是防 Redis 爆掉的最后一道保险。

### 4. Redis 宕机怎么降级

必须回答这个问题，不然方案是不完整的。

降级策略很直接：

- Redis 读失败：回源 MySQL
- Redis 写失败：临时直写 MySQL
- 热帖缓存失效：后台任务重新预热

核心原则只有一个：**缓存挂了可以变慢，但不能把业务一起带死。**

## 六、从点赞场景看 Redis 五大基本数据结构

这道题真正的主角是 `Set`，但它刚好能把 Redis 五大基本数据结构串起来。

| 数据结构 | 在点赞场景中的角色 | 为什么用它 |
|---------|------------------|-----------|
| `String` | 点赞数、访问量、阈值计数 | 简单、原子自增、自减快 |
| `Hash` | 热点帖子元信息 | 多字段聚合，便于局部更新 |
| `Set` | 某帖点赞用户集合 | 天然去重，适合关系判断 |
| `List` | 轻量异步队列或操作缓冲 | 只适合简单场景，不建议替代 MQ |
| `ZSet` | 热榜、推荐排序、延迟任务 | 需要按分数排序时最合适 |

### 1. 为什么不是所有东西都用 Set

因为数据结构是按职责选的：

- 要唯一关系，用 `Set`
- 要计数，用 `String`
- 要多个属性，用 `Hash`
- 要排序，用 `ZSet`

这才叫选型，不是“我会几个命令，所以全塞进去”。

### 2. 这题里最容易顺带被追问的延伸点

- 热榜怎么做？`ZSet`
- 活跃用户签到怎么做？`Bitmap`
- UV 怎么估算？`HyperLogLog`
- 简单异步队列怎么做？`List` 或 `Streams`

所以点赞题本身是一个很好的入口，能自然把 Redis 常见数据结构串出去。

## 七、这种场景题的回答思路怎么养

### 1. 先问自己三个问题

1. 这是个真问题还是臆想出来的？
2. 有没有更简单的方法？
3. 会破坏什么吗？

这三个问题能防止你一开口就上“分布式大礼包”。

### 2. 场景题统一答题框架

以后再遇到收藏、关注、评论、秒杀、抽奖这类题，都可以按这 5 步走：

#### 第一步：先定目标和约束

- 是高并发还是强一致？
- 是读多写少还是写多读少？
- 用户最在意的是快、准，还是绝对一致？

#### 第二步：拆核心数据

- 什么是事实数据？
- 什么是派生数据？
- 谁是主存储？
- 谁只是缓存或索引？

#### 第三步：讲主链路

- 写路径怎么走
- 读路径怎么走
- 热点怎么拦
- 数据怎么分层

#### 第四步：讲一致性和异常

- 重复请求怎么办
- 消息重复怎么办
- Redis 挂了怎么办
- 数据延迟能接受到什么程度

#### 第五步：最后再讲优化

- 缓存淘汰
- 排行榜
- 分库分表
- 监控与告警

顺序别反。先把主干讲清，再谈优化。

### 3. 常见扣分点

- 一上来先背 Redis 命令，不讲业务目标
- 不区分冷帖和热帖，默认全量缓存
- 只讲缓存，不讲数据库唯一约束
- 只讲高性能，不讲降级与容灾
- 只讲组件名字，不讲数据之间的关系

### 4. 怎么把这种思路练成习惯

最有效的方法不是死背答案，而是给自己强制套模板：

```text
目标是什么？
核心数据是什么？
主链路怎么走？
瓶颈在哪？
一致性怎么取舍？
异常怎么兜底？
```

每做一道题，都先口头回答这 6 句，再补细节。练多了，系统设计题就不会散。

## 八、面试里可以直接复述的收尾

如果面试官最后问“你总结一下”，可以这样收：

> 点赞场景本质上不是 Redis 题，而是数据分层题。  
> 我会先把点赞关系、点赞计数、热点信号和异步事件拆开，再决定哪些进 MySQL，哪些进 Redis。冷帖直连数据库，热帖进缓存；点赞关系用 Set 去重，计数和热度分开维护；通过 MQ 或定时任务异步落库，接受最终一致性；Redis 故障时降级回数据库。  
> 这套设计的核心不是组件堆砌，而是让热点走快路径、冷数据走便宜路径、异常场景也有兜底。
