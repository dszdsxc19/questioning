# 接口幂等性设计场景题

> [!tip] 方法论参考
> 本题涉及 API 设计和数据一致性，可参考 [[指导思想/从需求到上线——完整的系统设计方法论]] 中的 API 设计阶段和详细设计阶段。

## 题目描述

**请设计一个服务接口的幂等性方案。实现多次调用对系统产生的影响是一样的。以购物车场景为例，请完整梳理从进入结算页到支付完成的全链路核心交互。**

---

## 核心概念

### 什么是幂等性

> **幂等性（Idempotency）**：同一个请求，无论执行多少次，对系统状态的影响都是相同的。

| 操作类型 | 是否幂等 | 说明 |
|---------|---------|------|
| SELECT | ✓ | 查询不改变状态 |
| INSERT | ✗ | 多次执行会产生多条记录 |
| UPDATE | ? | 取决于更新逻辑（绝对值更新是幂等的，增量更新不是） |
| DELETE | ✓ | 删除多次结果相同（资源不存在也视为成功） |

### 为什么需要幂等性

1. **网络抖动**：用户重复点击、超时重试
2. **前端并发**：多个按钮同时触发
3. **支付回调**：第三方平台可能发送多次回调
4. **消息队列**：消费者重复消费消息

def generate_idempotent_token():
    # 生成全局唯一 Token
    token = generate_uuid()

    # 存入 Redis，设置过期时间（如 30 分钟）
    redis.setex(

### 4. 后端原子校验（Lua 脚本）

```lua
-- redis_idempotent_check.lua
local key = KEYS[1]
local token = ARGV[1]

-- 查询并删除的原子操作
local result = redis.call('GET', key)

if result == token then
    redis.call('DEL', key)
    return 1  -- 校验成功
else
    return 0  -- 校验失败
end
```

```python
def create_order(order_data, token):
    # 原子校验 Token
    script = """
    local key = KEYS[1]
    local token = ARGV[1]
    local result = redis.call('GET', key)
    if result == token then
        redis.call('DEL', key)
        return 1
    else
        return 0
    end
    """

    key = f"idempotent:token:{token}"
    valid = redis.eval(script, 1, key, token)

    if not valid:
        raise BusinessException("请勿重复提交")

    # ... 继续执行创单逻辑
```

**校验结果处理：**

| 场景 | Redis 状态 | 后端行为 |
|------|-----------|---------|
| Token 存在且匹配 | Token 存在 | 删除 Token，放行请求 |
| Token 已被使用 | Token 不存在 | 拦截，返回"请勿重复提交" |
| 非法 Token | Token 不存在 | 拦截，返回"请求无效" |
| Token 过期 | Key 不存在 | 拦截，返回"令牌已过期，请重新提交" |

## 第四阶段：支付并发控制（分布式锁）

### 9. 尝试获取分布式锁

```python
def initiate_payment(order_id, channel):
    # 以订单 ID 为 Key 尝试获取分布式锁
    lock_key = f"payment:lock:{order_id}"
    lock = redis.lock(
        lock_key,
        timeout=30,  # 锁超时时间
        blocking_timeout=0  # 非阻塞，获取失败立即返回
    )

    acquired = lock.acquire(blocking=False)

    if not acquired:
        # 锁已被占用，说明该订单的支付请求正在处理中
        raise BusinessException("支付处理中，请稍后再试")

    try:
        # ... 执行支付逻辑
    finally:
        lock.release()
```

**分布式锁设计要点：**

| 参数 | 说明 | 推荐值 |
|------|------|-------|
| Key | 订单 ID | `payment:lock:{order_id}` |
| 超时时间 | 防止死锁 | 30s（覆盖第三方支付接口耗时） |
| 阻塞策略 | 非阻塞 | `blocking=False` |
| 重试策略 | 前端提示用户 | 不自动重试 |

## 第五阶段：支付回调与状态同步

### 12. 异步回调处理

```python
@require_http_methods(["POST"])
def payment_callback(request):
    # 验证签名
    if not verify_signature(request.data):
        return {"code": 1, "message": "签名验证失败"}

    order_id = request.data.get('order_id')
    transaction_id = request.data.get('transaction_id')

    # 幂等性处理：先查询订单状态
    order = Order.objects.select_for_update().get(order_id=order_id)

    if order.status == 'PAID':
        # 已处理过该回调，直接返回成功
        return {"code": 0, "message": "处理成功"}

    # 更新订单状态
    with transaction.atomic():
        order.status = 'PAID'
        order.transaction_id = transaction_id
        order.paid_at = timezone.now()
        order.save()

        # 触发后续业务
        trigger_after_payment(order)

    return {"code": 0, "message": "处理成功"}
```

**回调幂等性关键：**

1. **先查后改**：先查询订单状态，已支付则直接返回
2. **行级锁**：`select_for_update()` 防止并发更新
3. **事务包裹**：保证状态更新和后续业务的原子性
4. **接口幂等**：第三方重复回调也能正确处理

## 防重策略总结

### 创单节点：Token 机制

| 保护对象 | 防重手段 | 存储介质 |
|---------|---------|---------|
| 库存扣减 | Token 原子校验（Lua 脚本） | Redis |
| 订单库 | Token 唯一性 + 数据库唯一索引 | Redis + MySQL |

### 支付节点：分布式锁

| 保护对象 | 防重手段 | 存储介质 |
|---------|---------|---------|
| 第三方支付调用 | Redis 分布式锁 | Redis |
| 订单状态更新 | 数据库行锁 + 幂等回调 | MySQL |

## 扩展问题

### 1. Token 机制与分布式锁的区别

| 对比维度 | Token 机制 | 分布式锁 |
|---------|-----------|---------|
| **使用场景** | 防止重复提交 | 防止并发执行 |
| **生命周期** | 一次性，用完即删 | 临时持有，用完释放 |
| **持有者** | 客户端（前端携带） | 服务端内部 |
| **典型场景** | 表单提交、订单创建 | 支付调用、库存扣减 |

### 2. 如何处理 Token 泄露？

- **绑定用户会话**：Token 与 User ID 绑定，校验时验证
- **缩短有效期**：设置为覆盖正常业务流程的最短时间
- **使用后立即失效**：Lua 脚本的查询并删除保证一次性
- **HTTPS 传输**：防止传输层被窃取

### 3. 分布式锁设置多长超时时间合适？

**原则：**
- 覆盖第三方支付接口的正常耗时（如 5-10s）
- 留有余量应对网络抖动（如 30s）
- 不能太长导致阻塞太久
- 不能太短导致业务未完成锁就释放

**最佳实践：**
- 设置合理的超时时间（如 30s）
- 实现锁的自动续期机制（Redis Redlock）
- 业务完成后主动释放锁

### 4. 如果支付回调丢了怎么办？

**多层保障机制：**
1. **前端轮询**：即使回调丢失，前端轮询也能感知到支付成功
2. **主动查询**：前端可点击"查询支付结果"按钮主动查询
3. **对账系统**：定时任务与第三方支付平台对账，补偿遗漏订单

## 参考代码实现

### Redis Lua 脚本完整版

```lua
-- idempotent_check_and_delete.lua
local key = KEYS[1]
local expected_token = ARGV[1]
local user_id = ARGV[2]

-- 获取当前 Token
local current_token = redis.call('GET', key)

if current_token == false then
    -- Token 不存在或已过期
    return {err = 'TOKEN_NOT_FOUND'}
end

if current_token ~= expected_token then
    -- Token 不匹配
    return {err = 'TOKEN_MISMATCH'}
end

-- 验证用户 ID（可选）
local owner = redis.call('HGET', key .. ':meta', 'user_id')
if owner and owner ~= user_id then
    return {err = 'TOKEN_OWNER_MISMATCH'}
end

-- 删除 Token
redis.call('DEL', key)
redis.call('DEL', key .. ':meta')

-- 校验成功
return {ok = true}
```
