## 页面渲染的完整流程

浏览器将代码转换为屏幕上可视像素的过程，主要包含以下核心阶段：

```
HTML/CSS/JS
    │
    ▼
┌──────────────────────────────────────┐
│ 1. 构建 DOM 树 (DOM Tree)            │
│    解析 HTML，生成节点树             │
└──────────────┬───────────────────────┘
               │
               ▼
┌──────────────────────────────────────┐
│ 2. 构建 CSSOM 树 (CSSOM Tree)        │
│    解析 CSS，计算样式规则            │
└──────────────┬───────────────────────┘
               │
               ▼
┌──────────────────────────────────────┐
│ 3. 生成渲染树 (Render Tree)          │
│    合并 DOM + CSSOM，过滤不可见元素  │
└──────────────┬───────────────────────┘
               │
               ▼
┌──────────────────────────────────────┐
│ 4. 布局 / 回流 (Layout / Reflow)     │
│    计算节点位置和几何尺寸            │
└──────────────┬───────────────────────┘
               │
               ▼
┌──────────────────────────────────────┐
│ 5. 绘制 (Paint)                      │
│    填充像素，生成图层                │
└──────────────┬───────────────────────┘
               │
               ▼
┌──────────────────────────────────────┐
│ 6. 合成 (Composite)                  │
│    合并图层，显示到屏幕              │
└──────────────────────────────────────┘
```

### 各阶段详解

| 阶段 | 说明 |
|------|------|
| **DOM 树** | 解析 HTML，将标签转换为节点，表示页面逻辑结构 |
| **CSSOM 树** | 解析 CSS，明确每个节点的样式规则 |
| **渲染树** | 合并 DOM 和 CSSOM，过滤掉 `display: none` 等不可见节点 |
| **布局 / 回流** | 计算每个可见节点在视口内的坐标位置和几何尺寸 |
| **绘制** | 将视觉属性（颜色、背景、边框等）转化为像素 |
| **合成** | 将多个图层按层叠顺序合并，最终显示到屏幕 |

## 回流 (Reflow)

回流指的是浏览器需要**重新计算页面元素的位置和几何形状**的过程。

### 触发场景

| 触发操作 | 说明 |
|----------|------|
| **DOM 增删移** | 添加、删除或移动可见的 DOM 节点 |
| **尺寸变化** | 修改 width、height、margin、padding、border |
| **内容变化** | 文字数量变化导致换行、图片加载完成撑开父元素 |
| **窗口调整** | 用户调整浏览器窗口尺寸（resize 事件） |
| **获取几何属性** | 调用 `offsetWidth`、`clientHeight`、`getComputedStyle` 等 |

### 强制同步布局

> ⚠️ **性能陷阱**：当 JavaScript 获取某些几何属性时，浏览器会强制清空渲染队列并立即执行回流。

```javascript
// ❌ 触发强制同步布局
function badPractice() {
  for (let i = 0; i < 100; i++) {
    const el = document.createElement('div');
    el.style.width = '10px';
    document.body.appendChild(el);
    // 每次循环都读取 offsetWidth，强制回流！
    console.log(el.offsetWidth);
  }
}

// ✅ 正确做法：分离读写
function goodPractice() {
  const els = [];
  for (let i = 0; i < 100; i++) {
    const el = document.createElement('div');
    el.style.width = '10px';
    document.body.appendChild(el);
    els.push(el);
  }
  // 批量读取
  els.forEach(el => console.log(el.offsetWidth));
}
```

### 性能评估

**回流是渲染链路中开销极大的操作。**

- 单一节点的回流可能引发**父级、子级、兄弟节点**的连锁回流
- 严重消耗计算资源，导致页面卡顿

## 重绘 (Repaint)

重绘指的是元素的**视觉外观发生变化，但未改变其空间位置和几何尺寸**时，浏览器重新涂抹样式的过程。

### 触发场景

| 触发操作 | 说明 |
|----------|------|
| **颜色变化** | 修改 `color`、`background-color` |
| **可见性** | 修改 `visibility`（hidden ↔ visible） |
| **视觉修饰** | 修改 `box-shadow`、`border-style`、`outline` |

### 性能评估

| 对比项 | 回流 | 重绘 |
|--------|------|------|
| **开销** | 极大 | 较小 |
| **是否改变布局** | 是 | 否 |

> **核心逻辑**：
> - 发生回流 **必定会** 触发重绘
> - 发生重绘 **绝对不会** 触发回流

```
┌─────────────────────────────────────────────┐
│              渲染代价金字塔                  │
│                                             │
│            ▲                               │
│           ╱ ╲                              │
│          ╱   ╲     回流+重绘               │
│         ╱─────╲    (布局+绘制)              │
│        ╱       ╲                           │
│       ╱─────────╲   重绘                  │
│      ╱           ╲  (仅绘制)               │
│     ╱─────────────╲                        │
│                                             │
└─────────────────────────────────────────────┘
```

## 底层原理深入

### 盒子模型与回流的关系

**是的，回流与盒子模型有直接且深刻的关系！**

```css
/* CSS 盒子模型构成 */
┌─────────────────────────────────────────┐
│           margin (外边距)                │  ← 影响兄弟元素位置
│  ┌───────────────────────────────────┐  │
│  │        border (边框)              │  │  ← 影响元素总尺寸
│  │  ┌─────────────────────────────┐  │  │
│  │  │     padding (内边距)        │  │  │  ← 影响内容区域
│  │  │  ┌─────────────────────┐    │  │  │
│  │  │  │   content (内容)    │    │  │  │
│  │  │  │                     │    │  │  │
│  │  │  └─────────────────────┘    │  │  │
│  │  └─────────────────────────────┘  │  │
│  └───────────────────────────────────┘  │
└─────────────────────────────────────────┘
```

**底层原因**：

1. **回流 = 重新计算盒子几何**
   - 回流的核心就是浏览器引擎重新计算每个元素的**盒子模型属性**
   - 包括：`content` 宽高、`padding`、`border`、`margin` 的计算值

2. **盒子模型属性的联动效应**

   | 修改属性 | 回流范围 | 原因 |
   |----------|----------|------|
   | `width/height` | 自身 + 子元素 | 内容盒子变化，子元素需重新布局 |
   | `padding` | 自身 + 内容区 | 内边距挤占内容空间 |
   | `border` | 自身 + 兄弟元素 | 边框宽度增加元素总尺寸 |
   | `margin` | 兄弟元素 | 外边距改变元素间距，影响流式布局 |
   | `box-sizing` | 全局 | 切换盒模型计算方式 |

3. **为什么回流代价极大？**

   ```
   元素 A 的 width 变化
       │
       ▼
   ├─ A 本身重新计算盒子尺寸
   ├─ A 的子元素需要重新布局（父容器变了）
   ├─ A 的后续兄弟元素需要重新定位（流式布局）
   ├─ A 的父元素可能需要重新计算高度（内容撑开）
   └─ 可能引发父元素的兄弟元素连锁反应
   ```

   这就是为什么单一元素的回流会引发**级联回流**。

### 渲染层与合成层

浏览器为了优化性能，将页面分层渲染：

```
┌─────────────────────────────────────┐
│           渲染层级结构               │
├─────────────────────────────────────┤
│                                     │
│  主线程 (CPU)                        │
│  ├── DOM 树                          │
│  ├── CSSOM 树                        │
│  ├── 渲染树 (Render Tree)            │
│  ├── 布局 (Layout) ← 回流发生在这里  │
│  └── 绘制 (Paint) ← 重绘发生在这里   │
│                                     │
│  ┌─────────────────────────────┐    │
│  │   绘制记录 (Paint Records)   │    │
│  └──────────────┬──────────────┘    │
│                 │                   │
└─────────────────┼───────────────────┘
                  │
                  ▼
        ┌─────────────────────┐
        │   合成器线程          │
        │  (Compositor Thread)  │
        │                      │
        │  ┌───────┐ ┌───────┐ │
        │  │ Layer1│ │ Layer2│ │  ← 合成层（可能在不同 GPU 纹理）
        │  └───────┘ └───────┘ │
        └──────────┬───────────┘
                   │
                   ▼
        ┌─────────────────────┐
        │      GPU 光栅化       │  ← GPU 加速在这里
        └─────────────────────┘
```

**关键概念**：

| 概念 | 说明 | 所在线程 |
|------|------|----------|
| **渲染层 (Render Layer)** | 拥有独立绘制上下文的层 | 主线程 |
| **合成层 (Compositing Layer)** | 拥有独立纹理的层，可 GPU 加速 | 合成器线程 + GPU |

**合成层创建条件**（Chromium 内核）：

```css
/* 以下情况会创建独立的合成层 */

/* 1. 3D 变换 */
transform: translateZ(0);
transform: translate3d(0, 0, 0);

/* 2. 显式声明 */
will-change: transform;
will-change: opacity;

/* 3. 视频、iframe、canvas 等特殊元素 */
video, iframe, canvas, WebGL

/* 4. opacity 动画（有 transition/animation 时） */
opacity: 0.5;
transition: opacity 0.3s;

/* 5. position: fixed/sticky（部分浏览器） */
position: fixed;

/* 6. filter 滤镜 */
filter: blur(10px);
```

### GPU 加速的底层机制

**为什么 `transform` 和 `opacity` 能避开回流？**

```
传统动画流程（触发回流）：
┌────────────┐    ┌────────────┐    ┌────────────┐    ┌────────────┐
│ JS 修改    │ →  │ 重新计算   │ →  │ 重新绘制   │ →  │ 重新合成   │
│ left/top   │    │ 布局 (CPU) │    │ (CPU)      │    │ (CPU+GPU)  │
└────────────┘    └────────────┘    └────────────┘    └────────────┘
   ❌ 每帧都回流      消耗大           消耗大

GPU 加速流程（合成层动画）：
┌────────────┐    ┌────────────┐    ┌────────────┐
│ JS 修改    │ →  │ 合成器线程 │ →  │ GPU 直接合成│
│ transform  │    │ 更新变换矩阵│    │ (显卡硬件)  │
└────────────┘    └────────────┘    └────────────┘
   ✅ 只在合成层操作    极快           极快
```

**底层技术**：

1. **纹理映射 (Texture Mapping)**
   ```
   合成层 → GPU 纹理 → 图形处理器直接变换
   ```

2. **硬件加速原理**
   - GPU 有数百上千个核心，专门处理并行图形运算
   - CPU 执行布局计算（复杂逻辑）
   - GPU 执行合成运算（简单但海量的像素操作）

3. **合成器独立线程**
   - 合成器运行在独立线程，不阻塞主线程
   - 即使主线程繁忙，动画仍可流畅运行

**Chrome 的分层架构**：

```
┌─────────────────────────────────────────────────┐
│                    浏览器进程                     │
├─────────────────────────────────────────────────┤
│  ┌─────────────────┐  ┌─────────────────────┐   │
│  │    渲染进程      │  │    GPU 进程          │   │
│  ├─────────────────┤  │  (GPU Process)       │   │
│  │ ┌─────────────┐ │  │                      │   │
│  │ │  主线程     │ │  │  OpenGL / Vulkan     │   │
│  │ │  - DOM      │ │  │  - Direct3D         │   │
│  │ │  - CSSOM    │ │  │  - Metal            │   │
│  │ │  - Layout   │◄─┼──┼────────────────────┼───┼──→ GPU 硬件
│  │ │  - Paint    │ │  │                      │   │
│  │ └─────────────┘ │  │                      │   │
│  │ ┌─────────────┐ │  │                      │   │
│  │ │ 合成器线程  │ │  │                      │   │
│  │ │ (Compositor)│─┼──┼────────────────────┼───┼──
│  │ └─────────────┘ │  │                      │   │
│  └─────────────────┘  └─────────────────────┘   │
└─────────────────────────────────────────────────┘
```

**`will-change` 的正确使用**：

```css
/* ✅ 正确：用于已知会频繁变化的元素 */
.animated-element {
  will-change: transform, opacity;
}

/* ❌ 错误：滥用会导致内存问题 */
* {
  will-change: transform;  /* 创建过多合成层，消耗大量内存 */
}
```

> **注意**：每个合成层都会占用 GPU 内存，滥用 `will-change` 会导致：
> - 内存占用飙升
> - 合成层过多时，合并层的开销反而超过性能收益

## 更优的工程实践

### 1. 利用合成层处理动画 (GPU 加速)

对于频繁变化的动画效果，**仅使用 `transform` 和 `opacity`**。

```css
/* ✅ 推荐：GPU 加速 */
.animated {
  will-change: transform, opacity;
  transform: translateX(100px);
  opacity: 0.5;
  transition: transform 0.3s, opacity 0.3s;
}

/* ❌ 避免：触发回流 */
.bad-animated {
  left: 100px;        /* 触发回流 */
  width: 200px;       /* 触发回流 */
}
```

**原理**：`transform` 和 `opacity` 在"合成"阶段处理，直接由 GPU 完成，**避开回流与重绘**。

### 2. 批量操作 DOM

```javascript
// ❌ 在循环中逐个修改 DOM
for (let i = 0; i < 100; i++) {
  const el = document.createElement('div');
  document.body.appendChild(el);  // 每次都触发回流
}

// ✅ 使用 DocumentFragment 批量操作
const fragment = document.createDocumentFragment();
for (let i = 0; i < 100; i++) {
  const el = document.createElement('div');
  fragment.appendChild(el);
}
document.body.appendChild(fragment);  // 只触发一次回流
```

### 3. 分离读写操作

```javascript
// ❌ 读写交替 → 强制同步布局
el.style.left = '10px';
console.log(el.offsetLeft);   // 强制回流
el.style.top = '20px';
console.log(el.offsetTop);    // 强制回流

// ✅ 先写后读
el.style.left = '10px';
el.style.top = '20px';
console.log(el.offsetLeft, el.offsetTop);  // 只回流一次
```

### 4. 其他优化技巧

| 技巧 | 说明 |
|------|------|
| **虚拟滚动** | 只渲染可视区域的 DOM，减少节点数量 |
| **CSS Contain** | 使用 `contain` 属性限制样式影响范围 |
| **避免 table 布局** | 表格中任何一个元素变化都会触发整个表格回流 |
| ** debounce/resize** | 对 resize、scroll 事件进行防抖处理 |

```css
/* 限制样式影响范围 */
.card {
  contain: layout style paint;
}
```

## 面试要点总结

### 核心概念

| 概念 | 定义 |
|------|------|
| **回流** | 重新计算元素位置和尺寸 |
| **重绘** | 重新绘制元素外观，不改变位置 |

### 性能关系

```
回流 ⟹ 重绘
重绘 ⇏ 回流
```

### 优化策略

1. **GPU 加速**：动画使用 `transform` 和 `opacity`
2. **批量 DOM**：使用 `DocumentFragment`
3. **分离读写**：避免强制同步布局
4. **减少影响范围**：使用 `contain` 属性

### 开发者工具定位

在 Chrome DevTools 中：

```
Performance 面板
  ├── Layout (回流)
  ├── Paint (重绘)
  └── Composite (合成)

主要关注：
  - 长时间的 Layout 任务
  - 频繁的 Paint/Composite 切换
```
